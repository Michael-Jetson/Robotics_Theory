# 概述

机器人是一个非常大的领域，足式或者足腿式机器人只是其中的一个小方向，足式机器人是模仿动物四肢形态和步态而设计的机器人，可以在复杂地形上自主行走、攀爬、越障，与轮式或者履带式移动机器人需要连续且结构化的地面的运动方式不同，足式机器人可以通过离散的、非连续的支撑点通过非结构化的地面，如在面对楼梯场景时，轮式和履带式机器人会难以应对，但是足式机器人却可以轻而易举的实现攀登，此外就是，足式机器人的机器人本体与其腿足结构的运动是解耦的，可以实现类似车辆主动悬挂的效果，并且可以通过规划足端的轨迹来跨越障碍物

一般足式机器人的一条腿有三个关节，侧摆关节、髋关节、膝关节，但是一般膝关节会通过连杆结构来将电机放置在其他地方，以便于降低腿部重量、提高运动灵活性，一个四足机器人总共12个关节对应12个电机

实际上足式机器人的运动控制问题是一个优化问题或者说规划问题，在非学习类方法或者说传统方法中，机器人的运动往往要基于某种指标，如路径最短、速度最快、能量损耗最小，并且要满足某些约束，如机器人加速度和速度有上限、某些空间无法到达并且不能涉足，所以一般会针对决策变量进行建模和构建约束方程，以此来求出满足约束的情况下代价最小的最优解，实际上在很多情况下，因为某些原因无法求出最优解，如无法完美建模、非线性问题难以求解等因此，可以将问题进行简化或者牺牲一部分解空间来换取问题的简化，如此哪怕最终无法求出真正意义上的最优解，也可以求出接近最优解的次优解，并且保证解是可行的

实际上上述求出的解往往代表着轨迹，这个轨迹会被发送给底层的跟踪器进行跟踪，并且还有观测器（IMU、轮速计等）用来实时估计机器人的当前状态（如位置、速度、加速度等）

当然足式机器人的运动规划问题会复杂一些，因为其与车辆或者无人机的运动模式有较大的区别，后面二者的运动是一种连续模式的运动，并且运动过程中的约束和运动模式几乎不会改变，因此可以构建统一的模型进行求解，但是足式机器人尤其是四足机械狗，其存在多种不同的运动模型，因此需要进行模式的切换

## 足式机器人发展史

### 硬件方面

足式机器人的发展离不开硬件的发展，就跟AI的发展离不开芯片的发展一样，缺少了高性能低成本的硬件设备支撑，足式机器人只会存在于实验室中而难以普及到各种领域，甚至只能做一些最基础的任务而无法发展到如今的盛况，对于基于力控的足式机器人来说，执行器的驱动方式主要是三种：液压驱动，电机驱动（带有扭矩传感器）和串联弹性驱动，这里主要是介绍前面两种

- 液压驱动：是早期机器人的主要驱动方式

    - 特点是功率密度大，可以输出巨大的力矩，并且不驱动关节运动时不消耗能量，这对于需要重型负载的机器人来说非常合适

    - 但是缺点就是无法进行准确的输出控制，并且带有非线性动力学和难以实现高频控制的缺点

    - 同时价格昂贵，并且维修保养的成本也较高，因此在后期逐渐被抛弃

- 电机驱动：电机驱动是目前最主流的驱动方式

    - 在更早期时，电机功率密度低，因此需要使用大减速比的减速器来输出大扭矩，但是这样会带来较大的摩擦力和转动惯量，进而导致无法输出可控扭矩，并且难以加入传感器进行扭矩观测

    - 现代电机克服了上述缺点，实现了高功率密度下的可控扭矩输出，并且可以实现高带宽控制，同时可以通过反驱动能力缓冲冲击

    - 在现代电机的加持下，低成本高性能四足机器人成为了可能，并且逐渐发展起来

不过现在也有电液混合驱动的机器人方式，可以结合液压的静止时无功耗的特性实现某些场景下的高负载，也就是可以使用较低的功耗来实现大负荷机器人的移动，虽然无法像纯电机四足一样进行高动态移动，但是相比之下可以大大降低同样载荷情况下的能耗，国内浙江大学等有团队研究该驱动方式，这里给出一个参考文献

### 模拟器方面

一个高效准确的模拟器是验证算法准确性的良好工具，如果缺少一个好的模拟器，那么就必须在真实机器人上直接进行实现，一方面会带来调试的复杂度，一方面则会加大实验成本，所以随着数值算法的发展，高效计算铰链刚体系统的正逆动力学成为了可能，相应的算法可以在毫秒级实现高维动力系统的准确计算与仿真，这十分有利于在模拟环境中执行算法实验来验证准确性

在早期，仿真环境通常使用弹簧和阻尼器模拟接触，也就是惩罚法，但是这种方法更适合模拟柔性接触而非刚性接触，如果提高系统的刚度接会导致计算成本的增加，因此很难在仿真环境下反应机器人与物理环境的接触问题

而随着图形学、计算机硬件的发展，基于 GPU 的模拟器发展了起来，诸如 Gazebo、Mujoco、Isaac Lab 等仿真器的产生，使得机器人的高精度模拟成为了可能，也为机器人控制算法与强化学习算法的发展提供了强大的工具，因为对于 RL 类方法而言，需要大量的采样（或者说试错）才可以训练出一个良好的策略，因此现代模拟器的大规模并行模拟能力保证了 RL 方法的可行性，目前的基于 GPU 的仿真模拟器（如 Isaac Lab）可以实现每秒数十万次的频率采集状态转换数据，极大地加速了机器人 RL 的训练速度

当然，也可以直接使用真实机器人进行训练和自主探索（或者直接使用真实机器人采集数据并训练），该范式可以保证训练数据的真实性并且保证在真实环境下的性能，但是成本高昂，并且存在机器人损坏的风险——在自主探索的初期，机器人会非常容易发生碰撞或者跌倒，但是目前也有一些额外机制方法来缓解安全问题，比如说安全感知强化学习、故障恢复或者基于模拟的预训练机制

### 控制算法发展

早期的足式机器人通常使用很简单的动力学模型进行建模与控制，比如说倒立摆模型、质心模型或者单刚体模型

当前，足式机器人的控制算法主要是分为两种，一种是基于优化的（Optimization\-Based），一种是基于强化学习的（Reinforcement\-learning）

- 优化方法通常会对机器人动力学进行建模，通过最小化成本函数的方法计算出未来一段时间内的最优控制量（一般是足端作用力）

    - 早期的控制策略是将问题描述为一个凸优化问题，最常用的方法就是二次规划问题，但是这种方法一般只针对当前状态进行优化，难以具备预测控制的特性，因此难以实现飞行相的运动策略执行，存在局限性

    - 后续基于 MPC 的方案成为了主流，该范式采用采用了接触规划分离的方法（也就是将接触问题和运动规划问题分离开来，降低问题的复杂性），并且结合了全身控制理论，实现了非常好的效果

    - 但是基于优化的方法难以扩展到更多的真实应用场景，比如说如何进行接触交互等不确定性问题，以及如何将感知信息融合进运动策略中，因此强化学习范式发展起来，以此来应对这些问题

- 强化学习方法会基于奖励函数来进行探索，通过最大化回报的方法实现最优策略

    - 足式机器人的 RL 方法的研究开始的较早，但是受限于各种条件，早期的模型参数量少且会使用手工设计，并且在硬件上进行微调，因此表征能力受限，后期使用了一些随机最优控制框架，比如说路径积分策略改进

    - 后面随着深度学习和神经网络的发展以及计算机硬件的演变，强化学习有了更强大的发展前景，也就是深度强化学习，DRL 方法依赖于在模拟环境中进行大量试错来进行学习，并且一旦学出最优策略，那么只需要进行简单推理即可实现快速控制（因为DRL的网络模型通常很小）

实际上两种方法都是基于某一种函数计算成本或者奖励，以此来实现运动策略的最优实现

# 机器人运动学

## 坐标变换

坐标变换是机器人领域中必不可少的概念，在足式机器人中一般有以下几种常用的坐标系

|坐标系名称|解释|
|---|---|
|世界坐标系|整个世界的坐标系，是一个固定的、绝对的坐标系，以此为基准描述机器人所在的世界|
|身体坐标系|通常会以机器人的某个部分（一般是基座或者底盘）为基准，描述机器人本体运动|
|关节坐标系|基于机器人的每个关节的位置和角度进行定义，一般会基于每条腿的三个关节和足端来建立关节坐标系|
|连杆坐标系|在每个连杆上定义一个相对连杆固定的坐标系，以描述每个连杆与相邻连杆之间的相对位置关系，实际上可以理解为本质上与关节坐标系相同，但是一般用来研究关节质心位置，用于机器人动力学的研究|

然后可以使用欧拉角、旋转矩阵、四元数等工具描述机器人的旋转运动，这几种表示本质上都是相同的，但是欧拉角一般只用于人机交互（很符合人的直观理解），其存在奇异性问题或者是万向锁问题故不适用于机器人姿态的长期描述，因为在特定情况下会丢失自由度并且无法插值和迭代，一般会使用四元数或者旋转矩阵描述机器人的位姿

那么如何描述坐标系的相对位姿关系呢，一个方法就是描述一个坐标系的基向量在另一个坐标系中的位置，这样就可以通过向量在一个坐标系中的坐标求出另一个坐标系下的坐标了，如果有两个坐标系，记为A、B，相对A坐标系而言，B坐标系的基向量就可以构成一个旋转矩阵，而其中B的基向量都是相对于A坐标系而言的，具体地，每个列向量的每个分量都是这个向量在A的各个坐标轴的投影的长度（也就是向量点积的概念）

然后，这个旋转矩阵还可以用于坐标变换，比如说同一个点或者向量在A、B两个坐标系下的坐标是不一样的，先记其在在两个坐标系下的表示为 $^BP$ 和 $^AP$ ，这两个表示实际上是同一个向量，那么就可以分别在两个坐标系下进行表示了，其中 $^AP_x$ 指的是在A坐标系下的x轴坐标，其等同于向量投影到A坐标系x轴分量的长度

这里要注意的是，一个向量的表示是唯一的，但是在不同坐标系下的坐标是不同的（不同坐标系的基向量也是不同的），某种意义上可以理解为存在一个绝对坐标系或者世界坐标系，其基向量就是绝对意义的正交单位向量，而一个没有表明在哪一个坐标系下的向量默认是在这个世界坐标系下的向量，向量的元素值就是其坐标，如果有一个在另一坐标系中的向量，那么就需要将其投影到这个绝对坐标系下查看其绝对表示，那么这个绝对表示就可能与其坐标向量有所不同

## 正向运动学

在机器人学中，正向运动学和逆向运动学是两个相对的概念，正向运动学是给的机器人的关节参数从而确定执行器的运动学参数，逆向运动学是给定末端执行器的目标位置和姿态，计算出所需的关节参数，正向运动学是由因推果，给定输入计算输出，逆向运动学是由果推因，通过输出推算输入

正向运动学的好处是计算简单且直接，通过变换矩阵或者几何计算就可以获取唯一解，逆向运动学则复杂很多，因为存在多解、无解和奇异解的特殊情况，所以需要更强大的方法进行求解

首先可以定义转轴或者关节（Joint）的概念，每个关节有一个自由度，可以绕某个轴（Axis）进行转动或者沿轴线位移，也就是分为转动关节和平动关节，然后就是连杆，连杆可以连接关节，并且认为是刚体；定义了关节和连杆的概念之后，就可以对机器人进行表示和建模了，这里使用的是DH参数法，D\-H参数法按照设定规则为每个连杆固连了一个坐标系，之后就可以方便的描述一个连杆坐标系到相邻的下一个连杆坐标系的转换关系。实质就是把相邻坐标系的转换分解为了若干个步骤，每个步骤均只有一个参量。这几个步骤对应变换的组合就完成了相邻坐标系的变换。

DH参数法实际上是一类方法，最早由D和H两人提出的原始DH方法称为标准DH法或者SDH法，由后人改进的DH法称为MDH法，二者的区别在于连杆坐标系建立的位置不同。SDH方法将连杆*i*的坐标系固定在连杆的远端，MDH方法把连杆*i*的坐标系固定在连杆的近端，同时执行变换的的顺序不同

SDH方法存在缺陷，所以现在基本上都是使用MDH方法或者MDH改进的方法表示

那么对空间中两个任意方向的轴，需要如何对其进行描述呢？根据空间几何的内容可以知道，存在一个线段与两个转轴都垂直（相对于两个转轴之间作垂线），这个垂线段的长度称为连杆长度；然后以公垂线为准，记录两个转轴的夹角角度，并且记为 $\alpha$，这个角度就是连杆扭曲角度（link twist），通过这两个参数就可以描述两个转轴之间的关系

如果多个连杆相互连接，那么就需要更多的参数描述多个连杆之间的关系，这里定义连杆顺序为，更靠近基座的连杆和关节的序号更小；第一个参数就是连杆偏移量或者是连杆偏距，也就是描述两个连杆与转轴交点之间的距离，第二个参数就是关节角度，也就是描述两个连杆的夹角角度（角度计算方法参考link twist的计算方法）

## 逆向运动学

在正向运动学中，通过确定关节向量（可能是位移或者角度）来确定末端的位置或姿态，逆向运动学是相反的过程，先确定末端执行器的位置，然后反向计算出不同的关节要如何转动或者移动才可以使得末端达到这个位置

首先有两个概念，就是Reachable Workspace（可达工作空间）和Dexterous Workspace（灵巧工作空间）

||可达工作空间|灵巧工作空间|
|---|---|---|
|定义|机械臂末端（通常取工具坐标系原点）在所有关节运动范围下能够到达的**所有点**的集合|机械臂末端可以用任何可能姿态达到的位置点的集合|
|特点|只考虑是否可以达到，不考虑以什么姿态达到|同时考虑姿态和位置，是可达工作空间的子集|
|直观理解|机械臂能够到的范围|这是一种“完全可控”空间区域，在那里机械臂的末端既能到达，又能指向任意方向|

所以在进行逆运动学的解算时，可以先判断目标位置是否在可达工作空间或者灵巧工作空间中，如果不在则表示没有可行解，就无需进行计算，如果在可达工作空间外则无解；在可达工作空间内但不在灵巧工作空间中时，解的数量有限（可能唯一解）；在灵巧工作空间内则通常有多解，比如说下图中的PUMA机械臂就是这种情况

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MjY2OTRlYjlmZjIyMzViYzk3MjA2MGJiMmRmYWI3OThfN2YzYmNlYmFmZTI4MTdhODdiMmNiNmFlOWE0YThkMDBfSUQ6NzU1MzExMDQzMDY3MzQ4NTgyNl8xNzgyMjEyNDk2OjE3ODIyOTg4OTZfVjM)

如果是有多解的情况，就需要考虑更好的解了，比如说离目前状态最接近的解，或者可以避开障碍物的解，或者最快速可以到达的解

求解的方式有两种，第一种是解析法，比如说基于代数的方法（针对某些特定结构时，可以直接通过解析解或者逆矩阵方法准确快速地求出，典型有六自由度机械臂）和基于几何的方法（如下图所示，基于空间几何和三角关系等得出解析解），第二种是数值法，然后还有优化方法或者神经网络方法

## 一阶微分运动学与静力学

正向和逆向的运动学目标是实现关节角度空间与机械臂末端（或者是足端）位置坐标空间之间的映射，而关节角度和末端位置的一阶微分分别是关节角速度和速度，因此就有了一阶微分运动学，同时也可以使用一阶微分运动学实现关节力矩和足端力的换算，这里最经典的方法就是计算雅可比矩阵，这里一种很简单的计算雅可比矩阵的方法就是直接求导，当然这种方法仅适用于自由度较低的机械臂等，如果要求解高自由度机械臂的雅可比矩阵，就需要借助旋量和伴随矩阵的相关方法

首先，根据机器人的正向运动学有下面的公式，$\mathbf x$ 是位姿向量（或者记为工作空间向量，包含位置和姿态，也可以仅含有位置），$\mathbf q$ 是关节角度（或者记为关节空间向量，也就是每个关节的角度等构成的向量），其中的正运动学函数是一个非线性映射

$x=f(q)$

因为四足机器人的腿部仅有三个电机也就是三个自由度，只能单独控制足端的位置或者姿态，但是对四足机器人来说腿部的姿态并不是那么重要，因此这里记录足端的位置为 $\mathbf p$，同时认为位置和角度都是时间的函数，故对两端求导，则有

$\mathbf{\dot p}
=
\begin{bmatrix}
\dot x\\
\dot y\\
\dot z\\
\end{bmatrix}
=
\frac{\partial \mathbf f}{\partial \mathbf q}\frac{\partial \mathbf q}{\partial \mathbf t}
 = 
\begin{bmatrix}
\frac{\partial f_1}{\partial q_1} & \cdots & \frac{\partial f_1}{\partial q_n} \\
\vdots & \ddots & \vdots \\
\frac{\partial f_m}{\partial q_1} & \cdots & \frac{\partial f_m}{\partial q_n}
\end{bmatrix}

\begin{bmatrix}
\dot{q_1}\\
\dot{q_2}\\
\dot{q_3}\\
\end{bmatrix}
\Longrightarrow 
\mathbf{\dot p}=\mathbf{J}(\mathbf q)\mathbf{\dot q}$

可以发现，这是一个线性表达式，其中的系数矩阵就是雅可比矩阵，其联系起了机器人末端速度与电机转速，而且在四足机器人的特殊关节限位下，雅可比矩阵是一个可逆矩阵，因此可以很方便地通过矩阵求逆来实现正向逆向的切换

当然，上述内容是机器人运动时的转换关系，在机器人静止地施加力时，就需要静力学的内容来进行求解，这时候整条腿的动能不变，也就是总功率为0，视为流入功率等同于流出功率，也就是关节对腿的做功功率等同于足端对地面做功功率，即有

$\tau_1 \dot{q}_1+\tau_2 \dot{q}_2+\tau_3 \dot{q}_3=F_x \dot x +F_y \dot y +F_z \dot z$

其中 $\tau$表示不同关节电机输出的扭矩或者力矩，F 表示足端对地面的力的矢量（通常向下），如果将上式化为矩阵形式则有

$\tau^T \cdot \dot q = F^T \cdot \dot p \Longrightarrow  \tau^T \cdot \dot q = F^T \cdot J \cdot \dot q \Longrightarrow \tau = J^T \cdot F$

可以发现，关节力矩和足端对地面作用力之间也是线性关系，也就是只要求出雅可比矩阵，就可以很方便地将其应用于运动学和静力学中

## 奇异性问题与零空间理论（WBC数学基础）

根据上面的推导可以获取工作空间速度与关节空间速度之间的线性映射关系，也就是雅可比矩阵，该矩阵是关节空间向量的函数，并且可以很容易发现，从关节到工作空间的正向速度映射是很容易实现的，对于一个明确的关节空间速度 $\mathbf {\dot q}$，必然存在一个明确的工作空间速度 $\mathbf{\dot x}$，并且其是唯一的，也就是只要确定了正运动学映射关系的情况下，每个关节的运动确定了，机械臂末端的位姿变化也就确定了，并且这个变化是唯一的

但是反过来却不一定，在欠驱动机械臂中，并不是所有的工作空间速度都可以有其对应的关节空间速度的，也就是其在物理上就无法实现故不存在；而对于冗余机械臂来说，其工作空间维度少于关节空间维度，对于一个工作空间速度，甚至有无数个关节空间速度与之对应。这个时候，我们可以定义 $\mathbf R(J)$ 为机器人在雅可比矩阵为 $J$ 时可以达到的工作空间速度的集合，其为 $R^M$ 的一个子集

**冗余 \(Redundancy\)** 意味着机器人的自由度（关节数 $N$）大于它需要完成的任务维度（工作空间维度 $M$），例如，一个7自由度的手臂（$N=7$）在3D空间中抓取一个物体（$M=6$，即3D位置\+3D姿态），这就是冗余

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=OGNmNmY5YzlmMGU1NGI0ZjAzNWJhNmI5YTg0ZmJlYjVfY2FjZWIyMWM5MGZhZGIzMzAwNDkyMDFiZTk4NTFiYzVfSUQ6NzU2NjUzMzY5NzE1NTE4NjY4OV8xNzgyMjEyNDk1OjE3ODIyOTg4OTVfVjM)

为描述这种冗余特性，这里使用了代数中的零空间理论来表示，定义 $N(J)$ 为雅可比矩阵 $J $ 的零空间投影矩阵，简称零空间矩阵， $N(J)$ 与 $J$ 满足如下特性，并且零空间矩阵的秩就是机器人的冗余程度：

$J\cdot N(J)=0$ 

零空间是一个输入的集合，其包含了哪些被矩阵映射为零向量的输入向量，其本质上是一个输入空间的子空间，其数学定义如下

$N(A)=\{w \in \mathbb R^N |Aw=0\}$

零空间的作用接在于其描述了矩阵方程的解的结构，也就是如果 $A \boldsymbol{w} = \boldsymbol{b}$ 有一个解 $\boldsymbol{w}_p$（称为**特解**），那么该方程的**所有解**（通解）都可以表示为：

$\boldsymbol{w}_{\text{general}} = \boldsymbol{w}_p + \boldsymbol{w}_0 \quad \text{, 其中 } \boldsymbol{w}_0 \in N(A)$

- $\boldsymbol{w}_p$（特解）：完成任务的那个“特定”输入。

- $\boldsymbol{w}_0$（零空间解）：你可以“免费”添加的、不会影响输出结果的“额外”输入。

特解 $\boldsymbol{w}_p$ 上**添加任意一个零空间向量 **$\boldsymbol{w}_0$**，输出结果 **$\boldsymbol{b}$** 保持不变！**如果对应工作空间和关节空间中的向量，那就是说，对于冗余机器人来说，其关节自由度大于工作空间自由度，也就是存在特殊情况：关节转动，但是机械臂末端没有位姿变化或者说保持静止，所以可以得出一个结论：只要机器人是冗余的（且不处于奇异状态），它的雅可比矩阵 $J$ **必然存在一个维度大于0的零空间 **$N(J)$，也就是对于一个工作空间中的任务 $\mathbf{\dot x}$ ，有如下的矩阵方程

$\dot{\boldsymbol x}=J\dot{\boldsymbol q}$

- **通解：** $\dot{\boldsymbol{q}}_{\text{general}} = \dot{\boldsymbol{q}}_p + \dot{\boldsymbol{q}}_0$，其中 $\dot{\boldsymbol{q}}_0 \in N(J)$

- $\dot{\boldsymbol{q}}_p$** \(特解\)：** 为了让机器人末端（手）以速度 $\dot{\boldsymbol{x}}$ 移动，关节必须执行的那部分速度。

- $\dot{\boldsymbol{q}}_0$** \(零空间解\)：** 机器人关节可以执行的“内部运动”。物理意义是机器人可以以某种方式驱动关节，而它的**末端（手）在工作空间中完全保持静止**（$\dot{\boldsymbol{x}}=\boldsymbol{0}$）

求解这种非齐次线性方程最简单的方法就是使用逆矩阵计算，但是对于冗余机器人来说，其雅可比矩阵并非方阵，并且是一个欠定方程组矩阵，不存在对应的逆矩阵，且有无穷多解，但是可以定义一个 $N\times M$ 的特殊广义逆矩阵 $J^+$，其满足 $JJ^+=\mathbb I$，称其为雅可比矩阵的右逆矩阵，特解可用这个**右逆矩阵 **$J^+$ 来计算，即 $\dot{\boldsymbol{q}}_p = J^+ \dot{\boldsymbol{x}}$ ，并且该矩阵对 $J$ 而言是最优且唯一的（至于为什么最优以及如何计算，会在后面推导）

在定义了该矩阵之后可以推出如下公式

$0=J-\mathbb I J=J-JJ^+ J=J(\mathbb I - J^+ J)$

因此零空间矩阵以及右逆矩阵可按如下公式求解：

$N(J)=\mathbb I - J^+ J \qquad J^+=J^T(JJ^T)^{-1}$

也就是说，对于一阶微分运动学的方程 $\dot{\boldsymbol x}=J\dot{\boldsymbol q}$ 可以直接获取其物理上最优的特解 $\dot{\boldsymbol q}_p=J^+\dot{\boldsymbol x}$

对于冗余机器人而言，在非奇异构型条件下，雅可比矩阵一定是行满秩的，$JJ^T$一定是可逆的，但是在实际工程中，当机器人接近奇异点时，$JJ^T$虽然理论上仍然可逆，但是其行列式接近0，会变成一个病态矩阵，计算其逆会导致数值溢出，所以一般会加以处理，比如说阻尼最小二乘法：$J^+=J^T(JJ^T+\lambda^2\mathbb I)^{-1}$，通过加入一个微小的阻尼项，可以保证矩阵始终可逆，即使在奇异点处也可逆，保证了算法的数值稳定性

接下来就可以使用零空间和广义逆矩阵的方法计算特解和零空间解了，二者的叠加就是通解，很容易得出通解有如下形式：

$\dot{\boldsymbol{q}} = J^{+}\dot{\boldsymbol{x}} + N\dot{\boldsymbol{q}}_{\forall}$

其中 $\dot{\boldsymbol{q}}_{\forall}$ 为任意满足 $\dot{\boldsymbol{q}}_{\forall} \in R^N$ 的关节空间速度向量，对上式等号两边同时左乘以雅可比矩阵 $J$ 可以很容易验证该公式的正确性

至此，在尽可能保证工作空间任务 $\dot{\boldsymbol{x}}$ 完成的前提下，关节可以按照 $\dot{\boldsymbol{q}}_{\forall}$ 的速度任意运动，来实现其他额外任务如关节避障等，像这种充分利用关节冗余特性，同时保证多个控制任务的控制方法称为全身控制（WBC）。

### 零空间理论数学证明

那么这个右逆矩阵是否是唯一的呢？因为冗余机器人的关节空间与工作空间的映射关系实际上是一个欠定方程组，所以肯定存在通解和特解，首先可以构建一个特解，使得方程成立，所以这个特解应该满足：$\dot{\boldsymbol{q}}_p=G\dot{\boldsymbol{x}}$，其中的矩阵 $G$ 实际上是一个有逆矩阵意义的 $N\times M$ 矩阵，并且需要满足：$JG=\mathbb I$ ，这里将满足此方程的矩阵 $G$ 称为矩阵 $J$ 的一个右逆矩阵，并且很容易发现，$J $ 存在无数个右逆矩阵，而 $J^+$ 是其中的一个特例，在数学上称为 **Moore\-Penrose 伪逆**

那么巨大数量的右逆矩阵中如何选取一个最优呢？可以通过能量最优的概念进行选取，也就是我们希望选取的右逆矩阵所计算出的关节转速的模长是最小的，这意味着可以通过最小的关节速度（或者说最节能和平滑的）来实现期望的工作空间速度，避免不必要的剧烈运动，因此可以定义代价函数或者优化目标，然后使用标准的**拉格朗日乘子法 \(Lagrange Multipliers\)** 来证明，该方法是一种求解“带约束优化问题”的经典方法：

$\min_{\dot{\boldsymbol{q}}} \left( \frac{1}{2} ||\dot{\boldsymbol{q}}||^2 \right) = \min_{\dot{\boldsymbol{q}}} \left( \frac{1}{2} \dot{\boldsymbol{q}}^T \dot{\boldsymbol{q}} \right)$

然后确定约束条件，也就是关节速度必须实现期望的工作空间速度

$J\dot{\boldsymbol{q}} = \dot{\boldsymbol{x}} \quad \Rightarrow \quad J\dot{\boldsymbol{q}} - \dot{\boldsymbol{x}} = \boldsymbol{0}$

我们引入一个 $M \times 1$ 的拉格朗日乘子向量 $\boldsymbol{\lambda}$ \(因为有 $M$ 个等式约束\)，构造拉格朗日函数 $\mathcal{L}$

$\mathcal{L}(\dot{\boldsymbol{q}}, \boldsymbol{\lambda}) = (\text{优化目标}) - \boldsymbol{\lambda}^T (\text{约束条件})\\[0.5em]
\mathcal{L}(\dot{\boldsymbol{q}}, \boldsymbol{\lambda}) = \frac{1}{2} \dot{\boldsymbol{q}}^T \dot{\boldsymbol{q}} - \boldsymbol{\lambda}^T (J\dot{\boldsymbol{q}} - \dot{\boldsymbol{x}})$

要找到最优解 $\dot{\boldsymbol{q}}$，我们必须找到 $\mathcal{L}$ 的平稳点 \(Stationary Point\)，即 $\mathcal{L}$ 对 $\dot{\boldsymbol{q}}$ 和 $\boldsymbol{\lambda}$ 的偏导数（梯度）都为零的点，因此分别求偏导

$\boldsymbol{0}=\nabla_{\dot{\boldsymbol{q}}} \mathcal{L} = \frac{\partial}{\partial \dot{\boldsymbol{q}}} \left( \frac{1}{2} \dot{\boldsymbol{q}}^T \dot{\boldsymbol{q}} - \boldsymbol{\lambda}^T J \dot{\boldsymbol{q}} + \boldsymbol{\lambda}^T \dot{\boldsymbol{x}} \right) = 
\dot{\boldsymbol{q}} - (J^T \boldsymbol{\lambda}) + \boldsymbol{0} 
\longrightarrow
\dot{\boldsymbol{q}} = J^T \boldsymbol{\lambda}\\

\boldsymbol{0} = \nabla_{\boldsymbol{\lambda}} \mathcal{L} = \frac{\partial}{\partial \boldsymbol{\lambda}} \left( \frac{1}{2} \dot{\boldsymbol{q}}^T \dot{\boldsymbol{q}} - \boldsymbol{\lambda}^T (J\dot{\boldsymbol{q}} - \dot{\boldsymbol{x}}) \right)=- (J\dot{\boldsymbol{q}} - \dot{\boldsymbol{x}})
\longrightarrow
J\dot{\boldsymbol{q}} = \dot{\boldsymbol{x}}$

现在获取了两个约束方程或者说等式，其中包含了两个未知数 $\dot{\boldsymbol{q}}$ 和 $\boldsymbol{\lambda}$

可以将等式一带入等式二中，因此有

$J (J^T \boldsymbol{\lambda}) = \dot{\boldsymbol{x}}\\[0.5em]
(J J^T) \boldsymbol{\lambda} = \dot{\boldsymbol{x}}$

现在，我们求解 $\boldsymbol{\lambda}$。这里我们假设机器人**不处于奇异状态**，因此 $J$ 是行满秩的， $M \times M$ 矩阵 $(J J^T)$ 是可逆的：

$\boldsymbol{\lambda} = (J J^T)^{-1} \dot{\boldsymbol{x}}$

将其代入等式一有如下最佳解向量，或者说可以直接计算出对应的伪逆矩阵然后就可以求出最小范数解：

$\dot{\boldsymbol{q}} = J^T \left( (J J^T)^{-1} \dot{\boldsymbol{x}} \right)\\[0.5em]
\dot{\boldsymbol{q}} = \left[ J^T (J J^T)^{-1} \right] \dot{\boldsymbol{x}}=J^+\dot{\boldsymbol{x}}\\[0.5em]
J^+=J^T (J J^T)^{-1}$

满足 $J\dot{\boldsymbol{q}} = \dot{\boldsymbol{x}}$ 约束的**唯一**最小范数解，正是由 Moore\-Penrose 伪逆 $J^{+}$ 给出的特解 $\dot{\boldsymbol{q}}_p$，并且其还有一个很好的性质，就是 $J^{+}\dot{\boldsymbol{x}}$ 位于$J$ 的行空间（与零空间正交），这是更深刻的数学原因，也是**后续相关推导**（零空间、多任务、全身控制）的**真正基石**。

在代数理论中有**秩\-零度定理 \(Rank\-Nullity Theorem\)**，也就是输入空间的维度等于值域空间维度加零空间维度，在机器人学中输入空间就是关节空间，值域空间就是工作空间，所以可以很容易得出冗余机器人的零空间的维度，而且在非奇异构型下零空间维度必定大于零，一定存在可以利用的内部运动 $\dot{\boldsymbol{q}}_0$，其不会影响机械臂末端的运动，也就是有

$J\dot{\boldsymbol{q}}_0 = 0$

那么如何使用一个具体公式表示这个零向量或者说内部运动，以此来实现规划或者控制等，或者说我们想引入一个**次要任务**，比如避障，这个次要任务可能希望我们执行一个任意的关节速度 $\dot{\boldsymbol{q}}_{\forall}$。我们不能直接用 $\dot{\boldsymbol{q}}_{\forall}$，因为它会干扰主任务，我们需要一个“过滤器”，也就是一个**投影矩阵 **$N$，它能将 $\dot{\boldsymbol{q}}_{\forall}$ “投影”到零空间 $N(J)$ 中，得到 $\dot{\boldsymbol{q}}_0$。

$\dot{\boldsymbol{q}}_0 = N \dot{\boldsymbol{q}}_{\forall}$

**投影后的向量通常无法完成“原始意图”**，它完成的是“在不干扰主任务的前提下，**最接近**原始意图的那个运动”，这也就是“优先级”控制的本质，因为投影之前的关节速度向量中的某些分量可能导致末端执行器的位姿变化，这会干扰并破坏高优先级的主任务，也就是说投影矩阵的作用就是过滤掉次要任务中可能导致影响主任务的部分，只保留安全的部分，也就是要求投影后的向量必须在零空间中

实际上零空间投影矩阵的意义**不是**完美地执行次要任务，而是**在完美地执行主任务的前提下，尽最大努力去近似次要任务**，而且其作为一个正交投影矩阵，其可以在零空间 $N(J)$ 中找到一个唯一的向量，使其与原始意图之间的欧式距离最小

这个矩阵必须满足，无论要执行的关节速度是多少，投影的结果一定位于零空间 $N(J)$ 中，根据零空间的定义，这有下面的公式，并且很容易发现，如果想让投影后位于零空间中向量最接近原始向量（或者说欧式距离最小），那么这实际上就是转化为了一个正交投影的问题，也就是投影后的向量必须是原始向量在零空间的投影，且误差向量 $(\dot{\boldsymbol{q}}_{\forall} - \dot{\boldsymbol{q}}_0)$ 与零空间本身垂直或者正交

$J \dot{\boldsymbol{q}}_0 = \boldsymbol{0} \quad \Rightarrow \quad J N \dot{\boldsymbol{q}}_{*} = \boldsymbol{0}$

因为给定的是任意关节速度，所以投影矩阵本身必须满足：

$J\cdot N =0$

也就是这个投影矩阵必须是一个雅可比矩阵的零空间矩阵，到这一步之后，很容易就可以根据伪逆矩阵的定义计算出零空间矩阵的数学表达式了，也很容易发现，这个零空间矩阵也是唯一的正交投影矩阵

# 机器人动力学

在MPC中，一般认为四足机器人的物理模型为质心模型（CoM模型），也就是机器人所有的质量集中在 机器人的躯体中心位置，以此来简化计算如果不进行简化，则每次MPC的计算都相当于对系统做多次仿真来计算最优控制量序列，会导致计算量暴增，同时为了接近这种近似模型，串联腿的四足机器人模型一般会将膝关节的驱动电机放置在髋关节处，搭配连杆来驱动膝关节，从而降低腿部的转动惯量，同时进行实时反馈和计算，不会损失过多的控制精度

## 足端作用力约束

足端的作用力约束有两种约束，第一种约束就是幅度约束，第二种约束是摩擦锥约束

幅度约束分为两部分，第一个部分就是认为摆动腿的足端作用力为0，这种很好理解，因为腿在空中不会与地面接触，自然不会有作用力产生并影响机身运动状态，至于摆动腿本身运动导致的对机身的作用力也认为是可以忽略的，故可以施加一个0约束

第二部分约束就是垂直方向的幅度限制，因为腿部电机是有物理极限的，其对腿部施加的力矩有限制，对应地，足端产生的对地作用力也有限制，因此需要在垂直方向上进行限幅：$f_i^z\leq f_{max}$，当然，有的地方也会加一个最小值约束

第二个约束是摩擦锥约束，为保证足底与地面不发生相对滑动，足底反力的水平分量不能大于其 竖直分量与滑动摩擦系数 $\mu$ 的乘积，即满足摩擦锥条件

$\mu \cdot f^z_i \geqslant \sqrt{(f^x_i)^2+(f^y_i)^2}$

# 控制理论基础

这个系统是个非线性系统，为了实现对非线性系统的控制，需要在平衡点附近对系统进行线性化，线性化后的系统就可以应用状态空间控制理论实现控制了。对于倒立摆模型来说，系统有两个平衡点，分别是角度为0和180度时，而系统并不关心小车的水平位置如何，并且角度为180度时为稳定平衡点，角度为0度时为不稳定点，而倒立摆系统的目标就是在不稳定点处实现稳定控制，因此主要关注此点

系统线性化的本质是在平衡点附近的系统状态变量微分方程采用一阶方程的方式来表达，类似于将方程采用泰勒公式展开后，忽略高阶项，只保留一次项的方式，从而完成线性化，使非线性系统变得线性，系统状态越靠近平衡点处，系统的线性化精度越高。系统线性化常用的方法为雅克比矩阵。

假设有一个非线性控制系统，其模型如下

$\dot x=f(x,u)$

该系统的平衡点为 $x=x^*$，则系统的线性化公式如下

$\dot x=J(x^*)x+J(u)u$

也就是说，系统最终化为了一个线性系统，可以使用如下表达式描述

$\dot X=AX+BU$

# 最优控制理论

## 概念

首先就是最优控制的概念，其研究动机是在约束条件下达到最优的系统表现，比如说物理被控对象存在各种物理极限，电机有转速和力矩极限，方向盘有转动极限，而最优系统表现也可以定义为误差均方的积分和控制输入均方的积分，前者代表追踪性能良好，后者代表输入小、能量损耗小，或者引入代价函数的概念（也是性能指标的概念），也即有SISO系统下的

$J=\int_0^t qe^2+ru^2dt$

其中 r 是权重系数，然后目标是寻找最优的控制输入u使得代价函数最小

如果基于状态空间的方法，则有

$\frac{dX}{dt}=AX+Bu\\
\\
Y=Cx\\
J=\int_0^tE^TQE+u^TRudt$

如果是离散域的控制模型，那么上式就会转化为累加形式

在实际意义中，每一个状态变量的重要性不同，比如说有时候更看重速度，有时候更看重力矩，所以就需要引入半正定矩阵来修改权重系数，也就是从原先的 $E^TE$ 变成了 $E^TQE$ ，其中的矩阵 $Q$ 一般是一个对角矩阵

当然实际应用中还需要考虑一些物理约束，比如说车辆的速度和加速度有极限，机械臂中的电机力矩有极限，这些约束都是不可突破的，都属于硬约束的范畴，与之相对的就是软约束，通过性能指标来约束某个变量不能太大（太大的话代价会过高）

## 模型预测控制理论

模型预测控制，即Model Predictive Control，其核心思想是通过模型来预测系统在未来某一段时间内的行为，然后通过在线计算获取最优控制输入，以此在满足约束条件的情况下最小化代价函数实现最优，这种方法与PID只考虑当前误差的控制方法有较大区别，因为一般用于数位控制，所以会使用离散型状态空间进行表达，首先是系统模型具体的表达形式如下，其中X是状态量，可以使用最小的一组量来准确描述系统，u是控制量或者说控制输入，Z是系统的输出（当然不一定是可以观测的输出），然后ABC三矩阵是系统矩阵，这里认为状态是已知的（或者可以使用观测器观测估计的）

$X_{k+1}=AX_k+Bu_k\\
Z_k=CX_K$

MPC的核心思想如下图所示，预测时域记为 f，控制时域记为 v，时域的概念就是范围的长度，在实际工程中时域也是可以调节的参数，其中的蓝线就是系统的状态或者控制输出，在当前时刻 k 以后的就是预测出的系统状态，而我们需要做的事情就是计算出一组控制输入来引导输出或者状态到我们所期望的轨迹

MPC中，一般默认预测时域大于控制时域，且控制时域之后的控制输入恒定（等于控制时域最后的控制量），这是因为MPC的核心是在线实时优化，所以需要在短时间内快速求解，如果控制时域过大会导致优化变量过多从而无法快速求解，所以就假定在控制时域之后的控制输入是固定值并且等于控制时域最后的控制量

我们使用 $u_{k+i|i}$ 来描述在 k 时刻计算出的 k\+i 时刻的控制输入，其他的系统状态和观测值等也采用类似的记法，然后以最小化期望轨迹和预测轨迹之间的误差为优化目标

$\mathbf{x}_{k+1|k} = A\mathbf{x}_k + B\mathbf{u}_{k|k}\\

\mathbf{z}_{k+3|k} = C\mathbf{x}_{k+3|k} = CA^3\mathbf{x}_k + CA^2B\mathbf{u}_{k|k} + CAB\mathbf{u}_{k+1|k} + CB\mathbf{u}_{k+2|k}$

将其进行拓展，拓展到控制时域内任意的时刻，也就是有如下的表达形式

$\mathbf{x}_{k+v|k} = A^v \mathbf{x}_k + \sum_{j=0}^{v-1} A^{j} B \mathbf{u}_{k+j|k}\\

但是这只是控制时域内的方程，一般情况下预测时域要大于控制时域，因为系统要看的更远然后进行当下决策，所以在控制时域之后和预测时域之间的区域也需要考虑进去，在这个范围内，控制输入是恒定的，也就是有

$\min_{\underline{\mathbf{u}}} \left\| \underline{\mathbf{z}}^d - \underline{\mathbf{z}} \right\|_2 = \min_{\underline{\mathbf{u}}} (\underline{\mathbf{z}}^d - \underline{\mathbf{z}})^T (\underline{\mathbf{z}}^d - \underline{\mathbf{z}})$

但是这种方法的问题在于无法约束控制输入的幅度，控制输入非常大以至于越过执行器的执行范围，导致实际应用不可行，或者可能引发执行器饱和，进而对反馈控制回路造成灾难性后果。因此，我们需要在代价函数中对控制输入增加惩罚函数。此外，还需在代价函数中引入误差函数，来使得系统尽可能收敛

惩罚项的定义如下，首先是最初输入的幅度，然后是不同输入之间的变化或者是差异，同时带有权重矩阵来修改惩罚项的系数，这种惩罚会对输入的幅度和变化率进行软约束，使其尽可能幅度小和变化速度慢

$\begin{aligned}
J_u & = u_{k|k}^T Q_0 u_{k|k} + \sum_{j=1}^{v-1} (u_{k+j|k} - u_{k+j-1|k})^T Q_j (u_{k+j|k} - u_{k+j-1|k})\\
&= 

\begin{bmatrix} 
u_{k|k} \\ 
u_{k+1|k} - u_{k|k} \\ 
u_{k+2|k} - u_{k+1|k} \\ 
\vdots \\ 
u_{k+v-1|k} - u_{k+v-2|k} 
\end{bmatrix}^T

\begin{bmatrix}
Q_0 & 0 & 0 & \ldots & 0 \\
0 & Q_1 & 0 & \ldots & 0 \\
\vdots & \vdots & \vdots & \ddots & \vdots \\
0 & 0 & 0 & \ldots & Q_{v-1}
\end{bmatrix}

\begin{bmatrix} 
u_{k|k} \\ 
u_{k+1|k} - u_{k|k} \\ 
u_{k+2|k} - u_{k+1|k} \\ 
\vdots \\ 
u_{k+v-1|k} - u_{k+v-2|k} 
\end{bmatrix}
\end{aligned}$

其中的控制差分部分可以进行简化

$\begin{bmatrix} 
I & 0 & 0 & 0 & \cdots & 0 \\ 
-I & I & 0 & 0 & \cdots & 0 \\ 
0 & -I & I & 0 & \cdots & 0 \\ 
\vdots & \vdots & \vdots & \vdots & \ddots & \vdots \\ 
0 & 0 & \cdots & 0 & -I & I 
\end{bmatrix}
\begin{bmatrix} 
u_{k|k} \\ 
u_{k+1|k} \\ 
u_{k+2|k} \\ 
\vdots \\ 
u_{k+v-2|k} \\ 
u_{k+v-1|k} 
\end{bmatrix} = W_1 \underline u
=
\begin{bmatrix} 
u_{k|k} \\ 
u_{k+1|k} - u_{k|k} \\ 
u_{k+2|k} - u_{k+1|k} \\ 
\vdots \\ 
u_{k+v-1|k} - u_{k+v-2|k} 
\end{bmatrix}$

则惩罚项可以进行化简

$J_u=(W_1 \underline u)^T Q (W_1 \underline u)={\underline u}^T W_1^T Q W_1 \underline u={\underline u}^T W {\underline u}$

对于跟踪误差部分也有类似的函数来描述差异，这个误差项描述了期望轨迹（或者目标轨迹）与实际轨迹之间的差别，并且也是总的代价函数的一部分

$\begin{aligned}
J_z &= (z^d_{k+1}-z_{k+1|k})^T P_1 (z^d_{k+1}-z_{k+1|k})+\cdots+ (z^d_{k+f}-z_{k+f|k})^T P_f (z^d_{k+f}-z_{k+f|k})
\\
&= (z^d-z)^TP(z^d-z)
\end{aligned}\\
其中P=
\begin{bmatrix}
P_1 & 0 & 0 & \ldots & 0 \\
0 & P_2 & 0 & \ldots & 0 \\
\vdots & \vdots & \vdots & \ddots & \vdots \\
0 & 0 & 0 & \ldots & P_{f}
\end{bmatrix}$

然后可以根据之前的系统输出方程，可以有

$\begin{aligned}
J_{\underline{\mathbf{z}}} &= (\underline{\mathbf{z}}^{d} - \underline{\mathbf{z}})^{T}P
(\underline{\mathbf{z}}^{d} - \underline{\mathbf{z}}) \\
&= (\underline{\mathbf{z}}^{d} - O\mathbf{x}_{k} - M\underline{\mathbf{u}})^{T}P(\underline{\mathbf{z}}^{d} - O\mathbf{x}_{k} - M\underline{\mathbf{u}}) \\
&= (\underline{\mathbf{s}} - M\mathbf{u})^{T}P(\underline{\mathbf{s}} - M\underline{\mathbf{u}})
\end{aligned}\\
其中\underline{s}=\underline{z}^d-Ox_k$

将两部分惩罚项进行叠加，就是完整代价函数，形式如下

$\min_{\underline{\mathbf{u}}} J=\min_{\underline{\mathbf{u}}} J_\mathbf{\underline z}+J_\mathbf{\underline u}=
\min_{\underline{\mathbf{u}}} \left( (\mathbf{s} - M\underline{\mathbf{u}})^T P (\mathbf{s} - M\underline{\mathbf{u}}) + \underline{\mathbf{u}}^T W \underline{\mathbf{u}} \right)$

将代价函数进行拆解可以有

$J=\mathbf{s}^T P \mathbf{s} - \mathbf{s}^T P M \mathbf{u} - \mathbf{u}^T M^T P \mathbf{s} + \mathbf{u}^T M^T P M \mathbf{u} + \mathbf{u}^T W \mathbf{u}$

求偏导，可有如下公式，这里注意一下，MPC的目标是计算出最优的控制输入，因此控制向量是自变量，其中P为对角矩阵，转置前后不变

$\frac{\partial s^T P s}{\partial u} = 0
\\[0.8em]
\frac{\partial s^T P M u}{\partial u} = M^T P s
\\[0.8em]
\frac{\partial u^T M^T P s}{\partial u} = M^T P s
\\[0.8em]
\frac{\partial u^T M^T P M u}{\partial u} = 2M^T P M u
\\[0.8em]
\frac{\partial u^T W u}{\partial u} = 2W u$

所以代价函数的导数为

$\frac{\partial J}{\partial \mathbf{u}} = -2 M^T P \mathbf{s} + 2 (M^T P M + W) \mathbf{u}$

因为代价函数是二次型，所以可以很容易的找出最优的向量，也就是导数为零时的变量，因此有

$(M^T P M + W) \mathbf{u} = M^T P \mathbf{s} \Longrightarrow \hat {\mathbf{\underline u}} = (M^T P M + W)^{-1} M^T P \mathbf{s}$

上述的过程，实际上就是根据系统状态方程和矩阵方法，将控制问题转化为一个QP问题然后使用各种QP求解器进行求解

# 四足运动控制算法

## VMC原理

虚拟模型控制算法，也就是VMC算法，是一种直觉控制方法，其核心思想是利用虚拟的机械部件（如弹簧、阻尼器、轴承，或者其他任何可以想象的组件）连接机器人内部作用点或者连接作用点与外部环境，产生虚拟力来驱动机器人实现运动，但是虚拟力不是实际执行机构的作用力或力矩，故需要通过雅可比矩阵将虚拟力转换到具体的执行机构上，从而计算出期望的关节力矩作为关节控制的输入，只不过真实关节的扭矩所产生力的效果与虚拟力产生的效果相同，均可驱使机器人按照期望运动。VMC的优势在于它简化了复杂动态系统的控制：用户可以直观地设计虚拟力，而不需要深入处理底层动态方程，从而使得传统方法难以描述的复杂任务可以用简单的虚拟模型轻松描述

例如希望机器人在一个未知的平面上施加冲击（如敲门），这实际上是一个很难使用数学描述的任务，但是如果使用VMC算法，只需要将虚拟的弹簧、阻尼器和一个有给定动能的虚拟质量附着在机器人手上，机器人的手在虚拟组件集合的作用下移动，对未知平面产生预期冲击之后，虚拟质量在虚拟弹簧阻尼器的作用下反弹了回来

再以足式机器人的二自由度单腿结构为例，其结构可简化为下图的形式，A、B两个转动副可以由电机驱动，对应的角度可以通过电机编码器测量出，两个连杆的长度 $L_1$ 和 $L_2$ 可以通过机械参数直接获取，控制任务主要是关注机构末端或者说足端 C 点的位置，位置可以使用直角坐标或者极坐标表示

VMC 方法是直觉控制方法中的典型代表，其特点是不需要进行机器人的逆运动学求解，综合考虑了机器人的机身高度、姿态和前进速度等，较为简单易控，极大提高了计算效率，控制效率较高。

VMC 的应用关键在于两点：一是在每个需要控制的自由度上构造恰当的虚拟构件以产生合适的虚拟力；二是在不同的相位状态利用相应的雅可比矩阵计算期望的关节力矩。

虚拟力并不是实际存在的作用力，而是假想的力，机器人的实际运动是靠各个关节的扭矩实现的，为了建立虚拟力与关节扭矩之间的关系，需要先将工作空间中的力或者力矩映射到关节空间下的关节力矩，这一步可以先通过正运动学模型获取空间映射关系，也就是将关节空间向量映射到工作空间向量上去

$\mathbf x=f(\mathbf q),其中\mathbf x=[x_1,\cdots,x_m],\mathbf q=[q_1,\cdots,q_n]$

其中的 x 是机器人坐标系相对于地面坐标系的 m 个自由度对应的位姿向量，这组向量必须完全描述系统的构型，在常用直角坐标系下可描述为坐标向量，q 为 n 个关节变量的位置向量，对于转动关节其对应关节的转动角度，对于移动关节其对应关节的位移

如果对其进行求导，就是雅可比矩阵，也就是一阶微分运动学和静力学部分的理论

注意一下，雅可比矩阵并不是一个固定不变的矩阵，其会随着关节角度的变化而变化

## 零力矩点ZMP

也称零倾覆力矩点，ZMP指的是地面上的一点，机器人本身的重力和惯性力所产生的合力矩在该点的水平分量为零，并且如果机器人运动时，ZMP位于机器人的支撑多边形内，可保证机器人运动过程的稳定性（或者说这是一种动态稳定性判据），这种方法相对于传统的静态稳定性判据（即重心投影法/CoG判据），充分考虑了机器人的重力和惯性力的合力的影响

机器人的支撑多边形是指四足机器人足底与地面之间所有接触点所构成的多边形区域，很容易想出，如果ZMP移动到支撑多边形以外，机器人就会开始倾倒

## 任务分层的WBC原理

全身控制，也即Whole Body Control，简称为WBC。为了降低运动控制算法的复杂性，控制任务通常会被分解为一组较为简单的任务，如保持平衡的躯干姿态控制，摆动相的足底轨迹跟随，或者用于防止滑动的接触力约束。每一个任务往往都需要多个驱动关节配合完成，同时每一个驱动关节又同时服务于多个任务，适应这种任务分解的理论框架是全身控制（WBC）。一般来说，为实现一组任务，所有驱动关节都能得到控制信号的控制系统都可以称作全身控制，但是四足机器人的全身控制至少实现了浮动基逆动力学。

因此，一个综合的机器人控制任务，可以依据不同的意义分解成不同的子任务，子任务可以通过加权平均的方式在同一个优先级上解决，也可以按照重要程度分成不同的优先级，低优先级任务不应该影响高优先级任务的执行。比如要用机械臂操作一个望远镜观察远处的星星，几十米的位置误差可能都不会对观测造成影响，但是不到 1°的角度误差就会有巨大的视野变化，因此在这种控制任务下，姿态任务的优先级要高于位置任务的优先级。这种带有优先级的全身控制方法在足式机器人上具有广泛的应用

那么如何进行WBC控制呢？回顾一下零空间理论，我们可以发现，对于冗余机器人来说，一个任务可以划分出特解和零空间解两部分，而这就是WBC的核心，因为这可以让机器人同时完成两件事：

- **任务1 \(高优先级\)：** 移动末端（手）以 $\dot{\boldsymbol{x}}_1$ 速度运动。（例如，保持杯子水平）。

- **任务2 \(低优先级\)：** 利用“内部运动”做点别的。（例如，避开障碍物）。

我们可以通过雅可比矩阵及其伪逆矩阵还有零空间矩阵计算出主次任务所对应的合适的关节空间速度，然后就可以在保证完成主任务的前提下尽可能执行次要任务，甚至可以扩展至多个任务优先级的全身控制

首先定义机器人的两个工作空间任务为 $\dot{\boldsymbol{x}}_1 $和 $\dot{\boldsymbol{x}}_2$，其雅可比矩阵和对应的零空间分别为 $J_1/N_1$ 和 $J_2/N_2$ ，并且任务任务1的优先级更高为了尽可能完成任务1，有

$\dot{\boldsymbol{q}} = \underbrace{J_{1}^{+}\dot{\boldsymbol{x}}_{1}}_{\text{特解 (完成任务1)}} + \underbrace{N_{1}\dot{\boldsymbol{q}}_{\forall}}_{\text{齐次解 (不影响任务1的内部运动)}}$

$\dot{\boldsymbol{q}}_{\forall}$ 是一个**任意**的 $N \times 1$ 向量，我们可以任意修改，并且在零空间矩阵的过滤下其不会影响任务1的执行，因此这也是尽可能满足任务2的控制按钮，我们希望改变该向量，使得 $\dot{\boldsymbol{q}}$ 也尽可能实现任务2的控制，因此根据上式可以有

$\begin{aligned}
\dot{\boldsymbol{x}}_2 &= \boldsymbol{J}_2 \dot{\boldsymbol{q}} \\
&= \boldsymbol{J}_2 \left( \boldsymbol{J}_1^+ \dot{\boldsymbol{x}}_1 + \boldsymbol{N}_1 \dot{\boldsymbol{q}}_\forall \right)\\
 &= \underbrace{J_{2}J_{1}^{+}\dot{\boldsymbol{x}}_{1}}_{\text{A. 任务1带来的“副作用”}} + \underbrace{J_{2}N_{1}\dot{\boldsymbol{q}}_{\forall}}_{\text{B. 我们可以控制的部分}} 
\end{aligned}$

因此，我们必须调整 $\dot{\boldsymbol{q}}_{\forall}$，让 B 部分去**补偿** A 部分的“副作用”，并最终等于 $\dot{\boldsymbol{x}}_2$，移项求解可得下面的最小范数解

$\dot{\boldsymbol{q}}_{\forall} = (J_{2}N_{1})^{+} (\dot{\boldsymbol{x}}_{2} - J_{2}J_{1}^{+}\dot{\boldsymbol{x}}_{1}) $

故可以得到最终的结果

$ \dot{\boldsymbol{q}} = J_{1}^{+}\dot{\boldsymbol{x}}_{1} + N_{1}  (J_{2}N_{1})^{+} (\dot{\boldsymbol{x}}_{2} - J_{2}J_{1}^{+}\dot{\boldsymbol{x}}_{1}) $

这里有一个文献证明了公式 $N_{1}(J_{2}N_{1})^{+} = (J_{2}N_{1})^{+}$，文献信息附上

> Maciejewski A A, Klein C A\. Obstacle avoidance for kinematically redundant manipulators in dynamically varying environments\[J\]\. The international journal of robotics research, 1985, 4\(3\): 109\-117\.
> 
> 

因此可以获取一个简化版本的公式

$ \dot{\boldsymbol{q}} = J_{1}^{+}\dot{\boldsymbol{x}}_{1} +   (J_{2}N_{1})^{+} (\dot{\boldsymbol{x}}_{2} - J_{2}J_{1}^{+}\dot{\boldsymbol{x}}_{1}) $

如果拓展到更多任务的全身控制，也就是有 $n$ 个控制任务，其中第 $i$个任务的工作空间位置使用 $x_i$表示，其雅可比矩阵与零空间矩阵分别是 $J_i$、$N_i$，默认 $i$从1开始计数，并且数值越小优先级越高

设 $\dot{q}_i$ 为按优先级实现前 $i$个任务的关节空间速度。当考虑第 $i$个任务时，可以将前 $i-1$个任务组合成一个大任务 $A_{i-1}$，即：

$\dot{x}_{i-1}^{A} = \begin{bmatrix} \dot{x}_1 \\ \dot{x}_2 \\ \vdots \\ \dot{x}_{i-1} \end{bmatrix} \quad
J_{i-1}^{A} = \begin{bmatrix} J_1 \\ J_2 \\ \vdots \\ J_{i-1} \end{bmatrix} \quad
\dot{x}_{i-1}^{A}=J_{i-1}^{A}\dot{q}_{i-1}$ 

其零空间矩阵为：

$N_{i-1}^{A}=\mathbb{I}-(J_{i-1}^{A})^{+}\cdot J_{i-1}^{A}$ 

假设此时 $\dot{q}_{i-1}$已知，参考关节空间向量的通解公式，可以有：

$\dot{q}_{i}=\dot{q}_{i-1}+N_{i-1}^{A}\dot{q}_{\forall}$ 

对该公式同时左乘 $J_{i-1}^A$可以验证 $\dot{q}_{i}$和 $\dot{q}_{i-1}$在任务 $A_{i-1}$所在的工作空间等效，现在尝试改变 $\dot{q}_{\forall}$，让 $\dot{q}_{i}$实现 $\dot{x}_{i}$的控制

$\begin{aligned}
\dot{x}_{i}&=J_{i}\dot{q}_{i}\\[0.5em]
&=J_{i}(\dot{q}_{i-1}+N_{i-1}^{A}\dot{q}_{v})\\[0.5em]
&=J_{i}\dot{q}_{i-1}+J_{i}N_{i-1}^{A}\dot{q}_{v}
\end{aligned}$ 

因此可以得出向量公式

$\dot{q}_{\forall}=(J_{i}N_{i-1}^{A})^{+}(\dot{x}_{i}-J_{i}\dot{q}_{i-1})$ 

带入之前的递推公式 $\dot{q}_{i}=\dot{q}_{i-1}+N_{i-1}^{A}\dot{q}_{\forall}$ 可以有

$\begin{aligned}
\dot{q}_{i}&=\dot{q}_{i-1}+N_{i-1}^{A}(J_{i}N_{i-1}^{A})^{+}\cdot(\dot{x}_{i}-J_{i}\dot{q}_{i-1})\\[0.5em]
&=\dot{q}_{i-1}+(J_{i}N_{i-1}^{A})^{+}\cdot(\dot{x}_{i}-J_{i}\dot{q}_{i-1})
\end{aligned}$ 

并且很容易得到起始条件为 $\dot{q}_1=J^+_1\cdot \dot{x}_1$ ，通过该递推公式，迭代**累加**所有 $n$ 个任务的、按优先级分配的运动分量，可得到满足所有任务的关节空间速度 $\dot{q}_{n}$ ，这实际上也是递推链的最终输出，也就是最终要发送给机器人的那个“总的”关节速度指令



## 优化的WBC原理

在基于任务优先级或者任务分层的WBC中，核心方法是利用了机器人的冗余性和零空间理论来进行分层，这与WBC的目标非常一致：在同一时刻，协调机器人全身的所有自由度，同时完成多个任务并且不违反物理约束。这种方式下的WBC在数学上非常清晰并且因为存在解析解所以计算速度很快，但是其过于依赖预先定义的任务优先级，也就是不同优先级的任务不会相互妥协，例如高优先级任务用尽了可用自由度的情况下，低优先级任务会完全失败，并且无法处理不等式约束，所以基于优化的WBC框架出现了，这是更加现代和通用的框架，其核心思想是将整个问题转化为一个约束优化问题，让求解器一次性找到最佳的相互妥协的解

这种方案的好处就是，可以通过调整不同任务的权重或者说重要性，使得WBC控制器对不同任务的关注度有所区别，并且可以使得不同任务在一定范围内相互协调，比如说最高优先级任务权重最高，但是如果允许其产生一点偏差使得第二优先级的任务的误差大幅度降低的话，优化器也可以实现不同优先级任务的妥协，而非为了完成高优先级任务但是不管低优先级任务，这种方法也就是**加权二次规划（Weighted Quadratic Programming, WQP）**

WQP 将所有 $N$ 个任务的误差项$e_i = \mathbf{A}_i \mathbf{x} - \mathbf{b}_i$组合成一个单一的加权最小二乘问题 

$\min_{\mathbf{x}} \quad w_1 \|\mathbf{A}_1 \mathbf{x} - \mathbf{b}_1\|^2 + w_2 \|\mathbf{A}_2 \mathbf{x} - \mathbf{b}_2\|^2 + \dots + w_N \|\mathbf{A}_N \mathbf{x} - \mathbf{b}_N\|^2$

这种方法实际上是一种软优先级的控制方法，其无法保证严格的优先级，一个较低权重的任务，如果其雅可比矩阵范数较大或者瞬时误差较大，仍然可能在优化过程中污染更高优先级任务的执行，并且权重系数也需要进行调整

当然还有一种更综合的方法，称为**层级二次规划（Hierarchical Quadratic Programming, HQP）**，HQP是WBC领域基于模型的“集大成者”，它巧妙地结合了NSP的严格优先级和WQP的不等式约束处理能力，将一个复杂的WBC问题分解为一系列（或称为“级联”）的**QP问题**，这些QP按照优先级的顺序依次求解，具体流程就是：先求解一个高优先级任务的QP问题，求出最优解，然后求解低优先级任务的QP问题，并且附带一个严格保持高优先级最优解的等式约束，也就是说，低优先级任务的QP问题只能在高优先级任务QP的零空间内优化求解

HQP方法继承了分层WBC和优化WBC的优点，保证了高优先级任务不会被低优先级任务破坏，并且可以很好的处理各种不等式约束问题，但是其需要连续求解多个QP问题并且要构建级联式求解，不论是在实现上还是在耗时上都要求更高

## 足端触地检测

四足机器人的足端触地检测是非常核心的一个功能，其提供了控制回路中的基本反馈信息，因为四足机器人的运动本质上是腿部在支撑相和摆动相之间循环切换的过程，这两种相位需要完全不同的控制律：

- **支撑相：**腿需要支撑身体，施加力以推动身体和对抗重力，控制目标是**力控制或阻抗控制**。

- **摆动相：**腿需要离开地面，沿着预定轨迹（如B样条）运动到下一个落脚点，控制目标是**位置控制或速度控制**。

如果没有准确的检测，就会导致控制逻辑混乱，从而产生异常的结果

- **脚已触地，但未检测到**：控制器认为脚在空中，继续执行位置控制，试图将脚移动到下一个点而不是输出作用力，但实际上脚已触地，这会导致：

    - 巨大的关节力矩输出，可能导致损坏电机或齿轮。

    - 机器人身体被意外撑起，破坏平衡。

    - 能量浪费。

- **脚在空中，但误判为触地**：控制器认为脚在支撑，对其施加支撑力来维持身体平衡，但实际上脚应该抬起或者落下，这会导致：

    - 腿的运动会改变方向，无法及时、准确地到达预定的落脚点

    - 因为无法正常落地，导致实际支撑的腿数量减少，可能导致摔倒

因此准确的触地检测非常重要，但是如何检测呢？如果直接使用力传感器，一方面可能造成额外的线路连接，另一方面，片式的力传感器很难覆盖较大的足底范围并且难以贴合球状的足底形状，并且这种薄片传感器的精度不足和带有噪声，如果使用高精度的力传感器则会造成成本剧增，因此大家都是选择状态估计的方法，也就是通过IMU、关节力矩等方法进行估计，一方面可以避免额外的传感器配置和线路连接问题，一方面可以实现更高效的动力学模型

### 概率融合触地估计

这个工作来源于下面的论文，这也是MIT Cheetah3相关的工作，可以实现99\.3的准确率和毫秒级的延时，作者提出该方法原本是为了非结构化地形上的触地检测，但是该方法实际上可以扩展到一般情况，因为在平地上时，四足机器人尚可以使用固定时间的调度器来维持运动，这种情况下可以认为机器人的运动模式固定，也就是每一步的时间都是一样的，时间到了自然可以认为触地了，但是在非结构化地面上，机器人必然会出现运动时间的偏移——更高的地面会导致腿部更早触地，而凹陷的地面会导致更晚触地，所以必须使用检测器来检测足端是否真正触地

> 
> De Luca, Alessandro, et al\. "Collision detection and safe reaction with the DLR\-III lightweight manipulator arm\." *2006 IEEE/RSJ international conference on intelligent robots and systems*\. IEEE, 2006\.
>

首先来了解一下背景，若要估计实际的触地状态，最直接的物理信号是足端的外部接触力。然而，MIT Cheetah 3 和许多其他足式机器人一样，并未在足端配备直接的力传感器。因此，必须仅通过本体感知（关节的扭矩、位置和速度）来估计这些力，解决这个问题的“直观”方法是比较“模型预期的扭矩”和“电机实际输出的扭矩”。但这种方法存在一个巨大缺陷：要计算出“腿部模型预期的扭矩”，你需要知道机器人腿部的**加速度（**$\ddot{q}$**）**。但在实际的数字控制器中，加速度通常是通过对速度信号进行数值微分得到的，这个过程会极大地放大噪声，并引入延迟，因此，任何依赖于加速度估计的力观测器，其信号都会非常嘈杂。

因此引入广义动量法（Generalized Momentum，GM），该方法的核心思想就是绕过对加速度的依赖，首先我们可以定义广义动量：$p=M(q)\dot q$，也就是惯性矩阵乘以关节速度

然后从机器人标准的动力学方程也就是拉格朗日方程开始，我们可以有：

$M(q)\ddot{q} + C(q, \dot{q})\dot{q} + g(q) =\tau+ \tau_{K}$

其中 $g$表示重力作用下的扭矩，而 $\tau$和 $\tau_{K}$表示作用在关节上的电机扭矩和未知的外部碰撞扭矩，也就是有：

$\tau_K = [M(q)\ddot{q} + C(q, \dot{q})\dot{q} + g(q)] - \tau$

很显然，其中带有加速度项，必须消去，这时可以对广义动量进行求导，发现所得结果中仍然含有加速度项

$\dot{p} = \frac{d}{dt}(M(q)\dot{q}) = \dot{M}(q)\dot{q} + M(q)\ddot{q}$

因此可以联合拉格朗日方程消去该项

$\dot{p} = \dot{M}(q)\dot{q} + [\tau+\tau_{K} - C(q, \dot{q})\dot{q} - g(q)]$

此时，我们利用机器人动力学中一个非常重要的特性，即 $\dot{M}(q) - 2C(q, \dot{q})$ 矩阵是**斜对称**（skew\-symmetric）的。这意味着 $\dot{M}(q) = C(q, \dot{q}) + C^T(q, \dot{q})$，因此带入有

$\dot{p} =\tau +\tau_{K} + C^T(q, \dot{q})\dot{q} - g(q)$

这个公式非常好的一点就是，其中并不包含加速度，仅仅依赖于总扭矩、位置和速度

## 步态解析

步态（Gait），实际上就是在一个周期内，各条腿的接触/摆动时序与相位关系的模式，也可以理解为是腿部的协调运动模式，这种模式是周期性的，可以实现机器人整体的运动与平衡

步态是周期性的，那么自然有步态周期的概念，一个周期就是一个完整的循环，从一条腿的支撑开始，到同一腿的下一次支撑，两次支撑时刻都时间差就是一个步态周期（也就是某个接触事件开始到下一次相同事件发生所经历的时间），通常用时间或相位（0到2π）表示，相应地，步态周期的倒数为步频

注意一下，步态与运动（特指移动运动）是分开的，并不是一个概念，步态只是一种协调运动的模式，而不是必须产生移动运动的模式，就好比人可以原地踏步，这时是没有位置上的运动的，并且人可以选择正常踏步、高抬腿踏步、大幅度手臂摆动踏步和行军式踏步等不同的腿部运动模式（实际上这就是步态的概念），人可以使用不同的步态（当然人的步态是带有手臂运动模式一起的步态）来进行移动，这对四足也是同样的，四足可以使用不同的步态进行移动，并且可以定义名义高度

因为步态是腿部的运动，所以会有腿部的抬起和落下，因此有了占空比β的概念，也就是每条腿在地面支撑的时间比例（β = 支撑时间 / 周期时间）。例如： 

- β \> 0\.5：静态步态（稳定，适合粗糙地形）。

- β ≈ 0\.5：动态步态（速度高，但需平衡控制）。

- β \< 0\.5：跳跃步态（有飞行相，所有腿离地）。

此外就是偏移量的概念，偏移量是指在一个步态周期中，每条腿支撑相开始时刻占总时间的比例。

在步态的方法中，一般会使用相位来描述步态，因为步态是周期性循环的，所以可以使用类似于钟表指针的相位概念来准确描述腿部的支撑和摆动时机，便于生成轨迹和多腿同步，并且相应的有时序图的概念，其中Trot和Walk两种步态的时序图如下，其中FR、FL、BR、BL分别是右前腿、左前腿、右后腿、左后腿， 黄色方块表示对应的腿处于支撑相，可见一个完整的步态由占空比和偏移量完整描述。

## 落足点规划

想象一下人类在跑步时，脚并不是随意落下的。为了向前跑，脚会自然地落在身体重心的前方；为了刹车，脚会落在重心后方；为了转弯，落点会偏向转弯的内侧。这个过程是下意识的，但背后遵循着精确的物理规律，如果没有一个好的落足点，那么机器人就无法实现良好的运动控制，甚至非常容易发生倾覆

四足机器人的落足点规划正是模仿了这一过程。其根本目标是：**通过主动选择落足点的位置，来产生期望的地面反作用力，从而控制机身的运动，并维持动态平衡。**最简单的落足点规划方法是Raibert启发式规划，这是一种在平面上行走的规划方法，也就是认为落足点总是在地面上，在计算落足点之前，需要获取控制指令（一般是期望的速度和角速度）、当前状态和步态参数，并且落足点规划一般在机身坐标系下进行，因为其只关心落脚点相对于自身的位置，计算完成之后将其转换到世界坐标系下，其具体的推理过程可以参考“Legged Robots that Balance”的2\.2\.3节等

在规划之前，需要先定义一些预知的参数，列表如下

|控制指令|线速度|$\mathbf v_d$|
|---|---|---|
||角速度|$\boldsymbol{\omega}_d$|
|当前状态|实际速度|$\mathbf v_{mes}$|
|步态参数|步态周期|$T_{period}$|
||支撑相时间|$T_{stance}$|
||摆动相时间|$T_{swing}$|

落足点规划中的第一个概念就是中性落足点，这个是机器人静止站立时刻每条腿的默认落足点，其通常位于对应髋关节正下方的地面上，就类似人类正常站立时的落脚点位置，并且使用机身坐标系描述，其定义如下

${^B\mathbf{p}}_{f,neutral}$

然后就是速度补偿项，这一项与当前速度有关，其是为了支撑机身在未来支撑相时间内的运动的，也就是落足点必须提前移动到未来的支撑中心，而这个补偿包含了两部分，其定义如下

$\Delta\mathbf{p}_{vel} = \underbrace{\frac{T_{stance}}{2} \cdot {^B\mathbf{v}}_{curr}}_{\text{前馈项}} + 
\underbrace{\sqrt{\frac{z_0}{||g||}} (\,{^B\mathbf{v}}_{curr} - {^B\mathbf{v}}_{des}\,)}_{\text{反馈校正项}} $

- **前馈项：**核心项，其预测了在**半个支撑相**时间内，机身将会移动的距离。为什么是半个支撑相？因为将脚落在未来支撑阶段的**中点**，可以使得整个支撑阶段的力最均匀，机身姿态最稳定。

- **反馈校正项：**用于修正误差。如果当前速度比期望速度快，这个差值为正，落足点会更靠前一点，起到“刹车”作用；反之，如果速度慢了，落足点会靠后一点，以产生更大的前推力。该公式来源于捕获点理论，其中的 $z_0$是行走时的名义高度

最后就是旋转补偿项，当机器人需要转弯时，身体会绕着垂直轴旋转。外侧的腿需要比内侧的腿迈得更远。这个补偿项就是为了实现这一点。

$\Delta\mathbf{p}_{rot} = \frac{T_{stance}}{2} \cdot ({^B\boldsymbol{\omega}}_{des} \times {^B\mathbf{p}}_{f,neutral}) $

此处的叉乘部分计算了由于期望角速度导致的位于中性落足点处的足端切向速度，同样，我们乘以半个支撑相时间 来计算落足点需要额外移动的位移，以平稳地支撑旋转运动

将上述的三项叠加起来，就得到了机身坐标系下的完整落足点计算公式了，然后就可以将其转换至世界坐标系下用于摆动相的腿部轨迹生成了

当然，上述的计算都是在假定机器人在平面上运动的情况下进行的，在真实世界中，这个位置可能是一个石块、一个水坑或者台阶的边缘，因此，规划区可以加入一个**落足点修正**过程，这个过程大致如下

1. **生成地形图**：使用激光雷达\(LiDAR\)或深度相机，实时生成机器人周围地面的三维高度图（Height Map）或代价图（Cost Map）。

2. **验证理想落足点**：检查由Raibert规划器计算出的理想落足点在地形图上是否安全（例如，是否平坦、没有障碍物）。

3. **搜索安全区域**：如果理想点不安全，就在其周围的一个小范围内（例如，一个半径为5\-10厘米的圆）搜索一个代价最低（最安全、最平坦）的替代点。

4. **更新目标**：将这个经过验证和修正后的安全落足点作为最终的结果

注意一下的是，这个公式实际上是对复杂动力学模型进行简化和线性化的结果，其结合了物理知觉、控制经验等内容，并且定义的落脚点并不是最优的，但是非常鲁棒和易于计算，如果想寻找出严格意义上最优的落脚点则需要求解复杂的非线性动力学方程

此外Raibert规划方法有一个很严重的缺点，在只有角速度控制指令不为零的时候，摆动腿的落足点所包含的补偿项实际上只有一个切向方向的补偿，根据叉乘的物理意义，其会将落足点沿着相对于机身的环形轨道的切向方向向外推动，导致期望的落足点相对于机身中心的距离增加（而理想情况是在仅旋转的情况下，期望的落足点应该在某圆形上，且圆形的圆心是质心在地面的投影位置），如果增加的幅度过大就会导致期望的落足点超出了足端可达区域，因此需要进行限制幅度等方法来约束

当然Raibert的规划公式是一种很早期的方法，后续有非常多的更新更优秀的方法，以及一些结合了感知信息的落足点寻找的方法，可以在非结构化的地形中寻找出合适的落脚点

## 控制框架

四足的经典控制方法中，MPC\+WBC的框架是非常常见的一种，MPC在质心与角动量方面优化出期望的质心轨迹和地面反作用力，WBC将这些期望和相应的约束转化为关节扭矩进行动力输出

四足机器人的控制框架实际上是一种混合控制的框架，因为在运动过程中，腿部会在摆动和触地之间来回切换，并且各有不同的控制模式，在摆动相时需要的是位置控制，在支撑相时需要的是力控，如下面的MIT Cheetah的框架框架图所示，控制指令（速度和角速度）输入之后，会生成对应的步态和支持力等，然后根据足端是否触地来选择控制模式并且计算出期望的关节力矩和转动角度等，并且使用PD控制器来快速控制，同时会使用卡尔曼滤波器实现对机器人进行状态估计和检测触地等

这里核心模块就是控制部分，这里是高层控制的内容，力控实际上是MPC计算出的合适支持力，摆动相控制实际上是使用轨迹优化方法计算出合适的摆动轨迹

足式机器人的MPC公式中，会有额外的约束，第一个就是线性动力学约束（也是状态转移方程），第二个就是摩擦锥约束，第三个就是转矩约束，这是一种针对电机转矩的约束，因为电机是有物理极限的，无法输出过大的力矩，所以哪怕计算出一个合适的足端作用力，如果其映射到关节上会导致某个电机超限，这也是不可行的，并且由于转矩和足端作用力之间通过带有正余弦函数的雅可比矩阵进行映射，所以这种约束是一种非凸约束，第四个约束是针对飞行相的双线性约束，四足机器人如果进行跳跃就会经历一个起飞阶段，这时起飞速度和飞行时间都是决策变量

对应转矩约束有如下形式

$||\tau||\leq \tau_{max}\\
\tau =J^T(q)\cdot F$

其中 q 通过逆运动学和足端位置计算得出，且有

$||J^T(q)\cdot F||^2\leq\tau^2_{max}\\[0.5em]
F^T \cdot J \cdot J^T \cdot F \leq \tau^2_{max}$

如此可以化为一个二次约束，其中的作用力 F 是决策变量

实际上四足机器人的控制框架是一种分层式的框架，会分为三层：

1. 高层就是步态规划层，会根据控制输入来确定步态的时序和落脚点的位置

2. 中层就是身体控制层，使用MPC方法优化出未来的身体轨迹和足端支持力

3. 底层就是关节控制层，通过逆运动学和WBC等，将MPC输出的轨迹和足端力转换为关节力矩和足端位置等

## 平衡控制模型

在足式机器人中，中层的控制的目标就是力和轨迹（或者说位置），分别对应了支撑相和摆动相的腿，其中摆动相的控制相对容易，因为通过落足点搜索或者规划的相关算法，可以为每条腿生成合适的落足点，而不同落足点之间的轨迹一般使用贝塞尔曲线或者B样条来连接，同时可能会使用QP方法进行局部优化，所以这里控制模型的核心就是力控，优化落足点位置的地面反作用力，并且因为机器人的四肢较轻，与整个身体相比惯性较低，所以可以忽略腿部的影响，以规划来自站立脚的地面反作用力，降低动力学建模的复杂度，这里主要的参考文献就是MIT的Cheetah

> MIT Cheetah 3: Design and Control of a Robust, Dynamic Quadruped Robot
> 
> 

首先定义动力学方程如下，该方程非常容易理解，F 为地面作用力矩阵，形状为 12x1，表示四条腿的三维作用力，如果是摆动腿则会加一个数值为0的约束（因为腿在空中不受力），具体的约束方法就是定义一个布尔变量来判断是否触地，触地的话设置为1，摆动的话设置为0，在最终约束不等式或者等式构建时就乘以对应的变量实现约束

然后整个方程描述的是在地面作用力影响下，机器人的加速度和角加速度的运动学约束，其中的 $[p_4 - p_c]_\times $表示叉乘的反对称矩阵，$p_i$表示足端在机身坐标系下的位置，$p_c$表示质心在机身坐标系的位置，$m$表示机器人的质量，$\omega_b$表示机器人的角速度，$I_G$表示在质心坐标系下的惯性张量

$\underbrace{
\begin{bmatrix} 
I_3 & \cdots & I_3 \\ 
[p_1 - p_c]_\times & \cdots & [p_4 - p_c]_\times 
\end{bmatrix} 
}_{ A}
F = A
\underbrace{
\begin{bmatrix} 
f_1\\
f_2\\
f_3\\
f_4\\
\end{bmatrix} }_{足端作用力}
=
\underbrace{
\begin{bmatrix} 
m(\ddot{p}_c + g) \\ 
I_G \dot{\omega}_b 
\end{bmatrix}
}_{ b}$

第一个控制器是平衡控制器，在论文中这是一个QP问题，平衡控制器的目标是输出一个使得机身运动尽可能跟随期望运动状态的地面作用力，也就是可以定义一个期望的运动状态（主要是角加速度和加速度），也就是有

$b_d=\begin{bmatrix} 
m(\ddot{p}_{c,d} + g) \\ 
I_G \dot{\omega}_{b,d} 
\end{bmatrix}$

由于模型支撑状态动力学模型是线性的，控制器可以表示为二次规划（QP）的解，也就是存在

$F^*=\min \limits_{F\in R^{12}}(AF-b_d)^T S (AF-b_d)+\alpha ||F||^2+\beta ||F-F^*_{prev}||^2\\[0.5em]
s.t. \quad CF\leq d$

最小成本函数反映了三者之间的权衡目标：将CoM动态驱动到期望目标的值，将使用的力最小化，使得当前作用力与上一次计算作用力之间尽可能接近（使得最终电机转矩变化不至于过于剧烈导致震动），并且约束项会将作用力限制在摩擦锥（或者摩擦金字塔）中保证物理可行

然后就是MPC的方案，因为单纯QP的方法只能考虑当下，无法对未来的一些目标状态（如跳跃）做准备，所以就需要MPC这种考虑未来的算法

NMPC 部分通过 OCS2 提供的 formulation 和求解接口，在每个周期都求解以下最优化问题：

$\begin{cases} 
\min_{u(t)} \phi(\mathbf{x}(t_f)) + \int_{t_0}^{t_f} l(\mathbf{x}(t), u(t), t)  dt \\ 

\text{s.t. } \mathbf{x}(t_0) = \mathbf{x}_0 \quad 初始条件\\ 
\quad \dot{\mathbf{x}}(t) = \mathbf{f}(\mathbf{x}(t), \mathbf{u}(t), t) \quad 非线性状态空间方程描述的系统动力学\\ 
\quad \mathbf{g}_1(\mathbf{x}(t), \mathbf{u}(t), t) = \mathbf{0} \quad 等式约束之状态控制耦合约束\\ 
\quad \mathbf{g}_2(\mathbf{x}(t), t) = \mathbf{0} \quad 等式约束之纯状态约束\\ 
\quad \mathbf{h}(\mathbf{x}(t), \mathbf{u}(t), t) \geq 0 \quad 不等式约束
\end{cases}$

对于足式机器人来说，我们有系统状态和输入分别为：动量（线动量和角动量）、广义坐标（机体位置、姿态、关节角度）和地反馈力、关节速度：

$x=[\mathbf h^T_{CoM},\mathbf q^T_b,\mathbf q^T_j]^T\quad
u=[\mathbf f^T_c,\mathbf v^T_j]^T$

而成本函数是简单的所有状态的跟踪误差和输入的二次成本，系统动力学使用了 centroidal dynamics，并有如下约束：

- 地反馈力在摩擦锥内；

- 触地腿不动；

- 摆动足端 z 轴位置满足由步态生成的曲线。

这个最优化问题被使用 Multiple Shooting 转换为 NLP 问题，并通过 SQP 求解，其中子 QP 问题通过近年来性能最强的 QP 求解器 [HPIPM](https://zhida.zhihu.com/search?content_id=214217348&content_type=Article&match_order=1&q=HPIPM&zhida_source=entity) 求解。

## 代码解析前言——硬件抽象与仿真

接下来就开始解析其代码，在解析核心控制代码之前会先对仿真交互部分进行解析，这部分是很重要的，因为对应的控制器（`legged_controllers`）并不“知道”自己是在和真实机器人还是在和仿真器打交道，它只和`ros_control`定义的标准接口通信，如果是真实硬件，则其会感知机器人真实状态和发送真实控制指令，如果是仿真，那么实际上对应的是`legged_gazebo`这个包，它“假扮”成真实机器人的硬件（或者说其是一个硬件抽象层，模拟了实物机器人的行为和反应），响应`legged_controllers`的请求，并与Gazebo的物理引擎进行数据交换。也就是说，`legged_controllers`并不会关心其所控制的对象是仿真还是实物，其只需要读取机器人对象的状态并且进行规划控制即可，这样就可以避免了仿真和物理之间的区别，不需要单独为两种情况单独编程，保证了复用性

那么如何实现一个这样的硬件抽象层呢？或者其是如何模仿机器人的行为与反应的呢？以自底向上的角度思考，机器人运动控制的对象就是关节电机，其会输出力矩驱动关节转动，进而使得腿部运动和输出力，同时机器人也会带有 IMU 和雷达等多种传感器来主动感知环境，同时我们也知道，Gazebo 这种模拟器非常擅长模拟机器人的物理行为，因此我们只需要重点研究如何构建一个可以在 Gazebo 中运动的四足模型并且控制其关节运动和感知其状态即可

### 硬件句柄与接口

句柄与接口实际上是一种资源管理的方式

首先看`legged_common`，该文件夹下定义了两个硬件的句柄\+接口，一个是触地检测句柄\+接口，一个是混合关节控制句柄\+接口，这样做是为了对单个硬件资源进行管理，因为查看底层代码可以发现，句柄类中都是以指针的方式进行数据管理的，这样可以实现控制器和硬件层之间的零拷贝实时数据交互并且便于做资源管理，也就是说

- 句柄：指向状态/命令缓冲区的指针或引用，外加安全检查，可以通过句柄读取状态和写入命令

- 接口：容器和资源占用管理（会管理所有同类型的`Handle`），可以实现句柄的注册与查找，并且实现资源声明与冲突检查

如此就实现了硬件抽象层的资源管理了，那么接下来就是资源的提供者和消费者了，资源的提供者可以是真实物理设备也可以是仿真设备，这里主要是介绍仿真部分，也就是`legged_gazebo`文件夹下的内容，该文件夹下实际上只有三部分内容，源代码、配置文件和世界模型，我们主要是解析代码

实际上，自定义的`HybridJointInterface`是对上层控制器暴露的复合自定义接口，因为现在四足机器人都使用的是PD混合控制器而不是单纯的位置/速度/力矩控制器，也就是上位机会发送一连串的带有多种信息的指令，但是`ros_control` 的任何一个标准接口都无法一次性传递这么多数据，所以定义了该接口，但是实际上机器人底层仍然是力矩控制`EffortJointInterface`，至于如何从混合控制到力控输出，就是实物机器人或者硬件抽象层实现的内容，因为不论是哪一种机器人对象，关节电机最终输出的还是力矩

### 硬件抽象

`legged_gazebo`代码中主要是定义了关节的数据、命令、IMU数据以及硬件仿真类`LeggedHWSim`，该类继承于一个默认的机器人仿真类`gazebo_ros_control::DefaultRobotHWSim`，用于在Gazebo中模拟真实机器人的行为和数据读取，这里普及一个知识点

- `RobotHWSim`：抽象基类或者接口类，定义了所有的Gazebo硬件仿真插件必须实现的契约，也就是几个关键的纯虚函数

    - `initSim()`：初始化插件，将 `ros_control` 的接口（如 `JointHandle`）与 Gazebo 中的模型（如 `gazebo::physics::Joint`）关联起来。

    - `readSim()`：从 Gazebo 仿真环境中**读取**状态（例如，关节的实际位置、速度、力矩），并将这些值写入 `ros_control` 的 `Handle`（句柄）中，以便控制器（Consumer）可以读取。

    - `writeSim()`：从 `ros_control` 的 `Handle` 中**读取**控制器（Consumer）写入的命令（例如，期望的力矩、位置），并将这些命令应用到 Gazebo 仿真环境中的关节上。

    - `eStopActive()`：用于检查紧急停止状态。

- `DefaultRobotHWSim`：是上述抽象基类的具体实现，对其中的标准功能进行了默认实现，用户无需自行编写

在这里，`LeggedHWSim`就是Gazebo中物理机器人的硬件仿真，如果查看源代码可以发现，其中包含了混合关节接口、接触传感器接口和IMU接口等资源，具体实现的功能如下

- **暴露自定义接口**：它向上层控制器（`LeggedController`）暴露 `HybridJointInterface`（混合关节接口）和 `ContactSensorInterface`（接触传感器接口）

- **连接 Gazebo 物理引擎**：它作为“胶水”，将 `ros_control` 的接口与 Gazebo 中的物理实体（关节、连杆）连接起来。

- **驱动仿真机器人模型**：其作为一个中间件，接受了控制器发出的控制指令之后进行解析，并且完成 PD 控制转换输出

- **IMU与接触状态读取**：其会读取Gazebo物理引擎中连杆的无噪声真值并且填入IMU数据中进行使用，接触状态的刷新也是通过Gazebo直接查看足端是否与其他物体发生碰撞来实现的，也就是整个仿真都处于一种非常理想化的情况下，其直接通过Gazebo获取完全准确的IMU数据和接触状态

那么如何使用该类呢？实际上`LeggedHWSim`并不是被直接运行的类，而是会作为一个插件被 `gazebo_ros_control` 动态加载和使用，整个过程分为四步：**编译时的“注册”**、**包的“声明”**、**URDF中的“指定”** 和 **运行时的“加载”**

首先是第一步，`LeggedHWSim` 作为一个 C\+\+ 类，首先需要向 `ros_control` 的插件系统“注册”自己，表明“我是一个可用的插件”，这一过程可以在`LeggedHWSim.cpp`的最后看到

```C++
PLUGINLIB_EXPORT_CLASS(legged::LeggedHWSim, gazebo_ros_control::RobotHWSim)
GZ_REGISTER_MODEL_PLUGIN(gazebo_ros_control::GazeboRosControlPlugin)  // Default plugin
```

这两行的作用分别是向ROS注册仿真插件和向Gazebo注册桥梁，以此来实现代码的应用

然后在`legged_gazebo`下可以发现一个`legged_hw_sim_plugins.xml`文件，该文件内容如下

```XML
<library path="lib/liblegged_hw_sim">
    <class name="legged_gazebo/LeggedHWSim" type="legged::LeggedHWSim"
            base_class_type="gazebo_ros_control::RobotHWSim">
        <description>
            The quadruped robot simulation interface which constructs joint handles from an SDF/URDF.
        </description>
    </class>
</library>

```

该文件的作用是，让ROS系统（Catkin）知道这个包导出了插件，这也是 ROS 的 `pluginlib` 和 `gazebo_ros_control` 查找插件的依据

这里注意一下，`imu.xacro`中并没有为对应的连杆附加一个IMU传感器插件来发布IMU话题数据，这是因为ROS话题通信会经历网络传输处理环境，这就意味着数据有延迟和抖动，并且是异步到达控制器的，这对于高频控制循环（数百赫兹）来说是无法接受的，而项目中使用的是这样的数据流：`Gazebo 物理引擎` \-\> `LeggedHWSim::readSim()` \(直接读取连杆状态\) \-\> `ImuSensorInterface` \(C\+\+ 共享内存\) \-\> `LeggedController::update()` \(直接内存访问\)

也就是数据没有经历ROS话题，其在 `ros_control` 的同一个 `read/update/write` 循环中被同步读取的。控制器在 `update()` 时拿到的 IMU 数据，与它拿到的关节角度、接触传感器数据是**完美对齐在同一时间戳**的。

这里总结一下 `robot.xacro` 中机器人命名规则或者说机器人的结构，如上图所示，七个文件都有各自的功能，但是其中需要重点关注的是 `leg.xacro` 文件，因为该文件定义了腿部结构与驱动插件，因此需要重点进行解析

那么在模型文件中是如何定义和构建这些的呢？因为前面提到 `leg.xacro` 文件中定义了腿部的模型，并且  `xacro` 语言是一种宏定义语言，可以实现类似函数一样的编程，也就是在 `leg.xacro` 中使用形参定义了一个腿应该如何建模，那么在更高级的文件中可以直接调用并传入参数生成不同的腿

那么新的问题来了，xacro 中定义了物理模型，在代码中是如何驱动和获取其状态的呢？比如说我们定义了一个四足机器人，那么其中的每个关节的运动状态是如何获取的？又如何将驱动命令发送过去的呢？插件（如 `LeggedHWSim`）被加载时，它只知道自己附着在整个机器人模型上，并不知道内部具体的关节或传感器对应的是哪一块仿真模型，这就需要一个链接或者映射的过程了，对于不同的硬件，实现方式有所不同

首先是驱动关节部分，这一块的内容在上面提到的`transmission.xacro`文件中，这个文件实际上是一个宏定义，其通过 `<transmission>` 标签实现映射，这也是最核心、最自动化的连接方式，`ros_control` 框架（以及 `DefaultRobotHWSim`）强制要求使用该标签来完成这个映射。

可以查看文件中是如何实现的，实现代码如下

```XML
<transmission name="${prefix}_HAA_tran">
    <type>transmission_interface/SimpleTransmission</type>
    <joint name="${prefix}_HAA">
        <hardwareInterface>hardware_interface/EffortJointInterface</hardwareInterface>
    </joint>
    <actuator name="${prefix}_HAA_motor">
        <hardwareInterface>hardware_interface/EffortJointInterface</hardwareInterface>
        <mechanicalReduction>1</mechanicalReduction>
    </actuator>
</transmission>
```

这些代码的解析如下：

- 定义一个名为 `${prefix}_HAA_tran` 的传动块（transmission），这个块描述“**关节\-执行器**”之间的映射规则，`gazebo_ros_control` 会据此注册相应的硬件接口

- 变速器模型选用 **SimpleTransmission**：一对一（1 关节对应 1 执行器）的**线性、常数齿比**传动模型，它只做比例缩放（由后面的 `<mechanicalReduction>` 指定），不考虑复杂耦合/非线性

- 指定被控制的关节名字为`${prefix}_HAA`（URDF 里有同名 `<joint>`），也就是表明在机械上是哪一个关节在动

    - `hardwareInterface`标签为被控关节**注册**（或“打上标签”）了一个硬件接口：`EffortJointInterface`（力矩控制接口）

    - `gazebo_ros_control`（`LeggedHWSim` 的父类）在 `initSim` 时会**扫描**整个 URDF，查找所有注册了 `EffortJointInterface` 的关节

    - `LeggedHWSim::initSim` 会通过 `ej_interface_.getNames()` 获取所有以这种方式“注册”的关节列表，然后将它们“包装”进自定义的 `HybridJointInterface`

    - 没有这一行，`LeggedHWSim` 插件就不知道它应该控制 "LF\_HAA" 这个关节

- 定义了与关节相连的驱动器（也是控制抽象）`${prefix}_HAA_motor`，并且声明执行器的接口也是关节力矩，这样传动层在转换中实现了单位一致，仅做比例缩放，并且后面参数指定了机械减速比为 1，`SimpleTransmission` 使用这个值来换算关节力矩和执行器力矩

### 仿真坐标关系与碰撞检测

在四足机器人中，坐标变换是很重要的，首先可以查看一下原项目中A1机器人的坐标变换，如下图所示，这种变换关系可以称为一种标准垂直构型，这种构型中膝关节位于髋关节的正下方（沿着 Z 轴方向），髋关节位于侧摆关节的左右方（沿着 Y 轴方向），这种关节构型虽然与真实机器人的关节位置存在一定差异，但是可以带来容易计算的好处

然后就是碰撞检测，在Gazebo中，对机器人进行碰撞检测是很重要的，你只有进行了碰撞检测，你才可以判断机器人是否出现问题——这也是Gazebo仿真的意义所在，其会尽可能准确的模拟机器人的真实运动状态，而当机器人与仿真环境中的其他物体发生碰撞的时候，Gazebo也会根据物理定律模拟真实碰撞的后果，但是碰撞检测有一个问题，就是Gazebo在最底层实际上是使用一系列的简单几何体基元来模拟更加庞大和外观多样的物体的，因此Gazebo这种模拟器的碰撞检测本质上也是通过判断简单几何体是否有交集的方法进行判断，那么就会引入一个新的问题：如何对几何形状复杂的物体做碰撞检查

大家可能会对具体方法有一个直观认知，只需要对每一个基元进行检测，判断其是否与其他物体发生碰撞就可以了，但是这种方法明显是低效的，因为一个机器人模型所需要的几何基元数量是异常庞大的，如果时时刻刻对如此众多的基元进行碰撞检测，那么即使有再大的算力都难以支撑，就会导致模拟器卡顿等情况，因此就需要想办法简化计算，如果我们查看原仓库中机器人的模型文件就可以发现，实际上其使用了一些简单的大几何体来进行碰撞检测，比如说对于机身，使用一个大的长方体来进行碰撞检测，即使机身与长方体并不是完全匹配，但是可以极大的降低计算量从而提高检测效率，因此机器人中普遍使用这种方法来进行近似

### 自定义机器人模型仿真

那么如何使用自己的机器人完成仿真以及后面的实物呢？首先你需要一个机械设计的大佬，为你在SolidWorks等机械设计软件中专门设计好模型结构，然后进行导出 URDF 文件，如此你就可以获取一个机器人模型文件了，至于如何设计，则是机械大佬的事情了；而关于仿真文件的结构，也建议模仿标配的方式，当然也可以把所有的结构合并到一个 URDF 文件中，只不过参考 xacro 宏的方式实现会更容易修改一些，然后方便起见，其中的关节命名方式建议与原代码相同，并且附带上相应的关节驱动控制器和硬件抽象层，以此实现仿真控制功能

这里注意一下，关节名称必须硬编码为 "LF\_HAA"、 "LF\_HFE"、 "LF\_KFE" 等，这是因为控制器代码 `LeggedController::init` 中已经进行了硬编码，如果想自定义关节名称会导致控制器无法控制关节输出力矩，因此 `xacro` 文件和控制器代码中对于关节名称的定义必须完全一致才可以正常工作，并且在使用SolidWorks设计时需要注意使用标准垂直构型

首先就是创建一个单独的存放模型描述文件的功能包，这里我们命名为 `legged_dm_description` 并将其与原始的宇树机器人模型文件存放的功能包放在同一个地方，同时也将我们自定义的机器人类型命名为 `dm`（实际上这里命名是可以自定义的，但是建议统一，因为后面控制器会根据机器人型号名去寻找某些文件），然后仿照原始的 `empty_world.launch` 文件写一个自己的 `launch` 启动文件实现启动，然后将机器人模型文件导出到一个统一的 URDF 文件中，并且将该文件使用机器人型号命名（比如说我们将其命名为 `dm.urdf`）并将其移动到 `/tmp/legged_control` 文件夹下，缺少了这个步骤就会报错如下，会导致控制器无法正常启动（因为控制器需要解析 URDF 文件获取机器人参数）

```Plain Text

[LeggedInterface] URDF file not found: /tmp/legged_control/dm.urdf
[ERROR] [1763102171.550279655, 54.902000000]: Initializing controller 'controllers/legged_controller' failed
```

## 代码核心解析——控制算法

项目采用的是目前足式机器人领域最主流、最高性能的控制架构之一：**级联式 Model Predictive Control \(MPC\) \+ Whole\-Body Control \(WBC\)**。

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=YTVkNzc5NmY3NTFiODg1MTlhNGVhZGQxYjRlZjE1Y2JfNzdiN2JjODE4NTUzNWUxODIxZTdiYjNjODMxMzVmZDNfSUQ6NzU3MTY2OTM2MzI5OTkyNjAxN18xNzgyMjEyNDk1OjE3ODIyOTg4OTVfVjM)

控制核心实际上涉及到四个文件夹，分别是`legged interface`、`legged estimation`、`legged wbc`以及`legged controllers`，并且最终的执行逻辑写在`legged controllers`的`LeggedController`类中，实际上可以将 `LeggedController` 理解为一个被动的类，其会以一种标准，定义了一系列回调函数（`init`, `update`, `starting`, `stopping`）。而 `ControllerManager`** **则是那个主动的执行者，其在一个由硬件/仿真驱动的高频实时循环中，在正确的时间、按正确的顺序，负责去调用这些回调函数。

那么 `ControllerManager` 在每个控制周期都会严格按照以下顺序执行

1. `robot_hw->read(time, period)`：

    - 硬件接口（`HardwareInterface`）从真实硬件或 Gazebo 读取所有最新的传感器数据（IMU 读数、关节角度、足底接触力等）。

2. `controller_manager->update(time, period)`：

    - 这是最关键的一步，`ControllerManager` 会遍历它管理的所有已启动的控制器（包括 `LeggedController` 实例）。

    - 然后，它会主动调用 `LeggedController::update(time, period)` 函数。

    - 这就是 `update` 函数被执行的地方，所有核心控制逻辑（状态估计、WBC 计算）都在这个 `update` 调用中同步执行。

3. `robot_hw->write(time, period)`：

    - 在 `update` 函数内部，代码调用了 `hybridJointHandles_[j].setCommand(...)`。这并没有立即发送命令，只是将命令*写入*了 `robot_hw` 的一个缓冲区。

    - 在 `update` 完成后，`ControllerManager` 会调用 `write()`，此时硬件接口才会真正将缓冲区中的所有命令（位置、速度、前馈力矩）打包发送给电机。

### 控制器基类与生命周期规范

前面我们提到，`LeggedController`类中定义了控制器的具体实现与执行逻辑，该控制器是继承`ros_control`的基类 `controller_interface::MultiInterfaceController` 来实现的，如此操作可以将算法控制器做成插件，统一了控制器如何拿硬件句柄、怎么被加载/启动/更新/停止，并与资源管理、控制器切换配合。

实际上该基类是一个模板基类，可以根据所需要的硬件接口来选择和实现，比如说源代码中是如此定义的

```C++
class LeggedController : public controller_interface::MultiInterfaceController
<HybridJointInterface, hardware_interface::ImuSensorInterface,ContactSensorInterface>
```

也就是该控制器同时需要混合关节控制，IMU和接触传感器三种接口，然后我们会根据其四大核心回调函数来进行讲解分析

#### `init`函数

这是加载配置的函数，其会在`ControllerManager` 加载该插件时执行一次，核心任务是为后续任务做好准备

注意一下，在这里硬编码了关节名称，所以如果想使用自定义的模型文件进行仿真，也需要使用对应的关节名称来与之匹配，否则会导致无法控制

```C++
/*
RobotHW 是控制器获取所有硬件资源（关节、IMU、传感器）的唯一途径
controller_nh 用于从控制器私有命名空间中抓取参数
*/
bool LeggedController::init(hardware_interface::RobotHW* robot_hw, ros::NodeHandle& controller_nh) {
  // Initialize OCS2
  std::string urdfFile;
  std::string taskFile;
  std::string referenceFile;
  controller_nh.getParam("/urdfFile", urdfFile);
  controller_nh.getParam("/taskFile", taskFile);
  controller_nh.getParam("/referenceFile", referenceFile);
  bool verbose = false;
  loadData::loadCppDataType(taskFile, "legged_robot_interface.verbose", verbose);

  // 创建 leggedInterface_ 对象，这是OCS2的问题中心，通过解析URDF和taskFile，构建了最优控制数学模型
  setupLeggedInterface(taskFile, urdfFile, referenceFile, verbose);
  // 创建SqpMpc求解器、动力学转换工具，并设置步态接收器和参考指令接收器
  setupMpc();
  // 创建MPC-MRT接口并启动**MPC**后台线程，这是因为MPC频率远低于WBC（也就是update函数的内容），所以需要单独线程
  setupMrt();
  // 可视化与运动学
  ros::NodeHandle nh;
  CentroidalModelPinocchioMapping pinocchioMapping(leggedInterface_->getCentroidalModelInfo());
  //创建了一个末端执行器（足端）运动学的求解器，用于计算雅可比矩阵及其导数
  eeKinematicsPtr_ = std::make_shared<PinocchioEndEffectorKinematics>(leggedInterface_->getPinocchioInterface(), pinocchioMapping,
                                                                      leggedInterface_->modelSettings().contactNames3DoF);
  // 创建了用于Rviz可视化的发布器，实现MPC期望轨迹、WBC计算力矩、状态估计等算法的结果可视化
  robotVisualizer_ = std::make_shared<LeggedRobotVisualizer>(leggedInterface_->getPinocchioInterface(),
                                                             leggedInterface_->getCentroidalModelInfo(), *eeKinematicsPtr_, nh);
  selfCollisionVisualization_.reset(new LeggedSelfCollisionVisualization(leggedInterface_->getPinocchioInterface(),
                                                                         leggedInterface_->getGeometryInterface(), pinocchioMapping, nh));

  // Hardware interface
  auto* hybridJointInterface = robot_hw->get<HybridJointInterface>();
  // 这里硬编码了关节名称
  std::vector<std::string> joint_names{"LF_HAA", "LF_HFE", "LF_KFE", "LH_HAA", "LH_HFE", "LH_KFE",
                                       "RF_HAA", "RF_HFE", "RF_KFE", "RH_HAA", "RH_HFE", "RH_KFE"};
  for (const auto& joint_name : joint_names) {
    hybridJointHandles_.push_back(hybridJointInterface->getHandle(joint_name));
  }
  auto* contactInterface = robot_hw->get<ContactSensorInterface>();
  for (const auto& name : leggedInterface_->modelSettings().contactNames3DoF) {
    contactHandles_.push_back(contactInterface->getHandle(name));
  }
  imuSensorHandle_ = robot_hw->get<hardware_interface::ImuSensorInterface>()->getHandle("base_imu");

  // State estimation
  setupStateEstimate(taskFile, verbose);

  // Whole body control
  wbc_ = std::make_shared<WeightedWbc>(leggedInterface_->getPinocchioInterface(), leggedInterface_->getCentroidalModelInfo(),
                                       *eeKinematicsPtr_);
  wbc_->loadTasksSetting(taskFile, verbose);

  // Safety Checker
  safetyChecker_ = std::make_shared<SafetyChecker>(leggedInterface_->getCentroidalModelInfo());

  return true;
}
```

## 代码应用解析——自定义仿真四足部署

在前面的一系列解析之后，我们就可以开始应用了，这里我们使用自行构建的四足机械狗进行应用（包括仿真和实物部署），也就是从零开始搭建，这一部分主要是控制器与电机之间的接口实现，前面的硬件抽象与仿真章节中层提到过，控制器并不知道是跟真实硬件还是仿真模型打交道，其只是与硬件接口 `hardware_interface` 打交道，因此实现最终硬件实物控制的关键就是做好硬件接口对象，让控制器的指令可以准确及时的转换到关节电机上并实现力矩的输出，并且可以及时读取关节状态与 IMU 的数据并放入缓冲区供控制器读取从而进行计算，在原仓库中，对应的代码是 `legged_hw` 文件夹

### URDF文件转换

首先从SolidWorks中将机器人模型导出，获得一个URDF的描述包，这里将其命名为dmgo\_description

“模型文件”被同时用于控制侧的 Pinocchio 动力学建模和仿真侧的 ros\_control/Gazebo 硬件抽象，两条链路分别从URDF 文件路径和ROS参数服务器上的 URDF 字符串读取，而且控制器还对关节/足端命名与顺序有硬编码假设，因此需要格外注意三点

- URDF 中要设置 `<transmission>`，Gazebo ros\_control 才可以创建 `EffortJointInterface `句柄，`legged_gazebo/LeggedHWSim` 才会拿到关节控制接口

- 模型有 Gazebo 插件块（`gazebo_ros_control` \+ `p3d_base_controller`），`legged_control` 的仿真接口才会加载

- IMU和关节

主要是有如下属性

- 摩擦系数mu1 / mu2

    - 没设置时 Gazebo 会用默认摩擦，通常会导致脚端“打滑”或行为不稳定

    - 摩擦用的是各向异性库仑摩擦模型，所以要给两个方向的摩擦系数，但是默认各向同性即可，也就是统一

- self\_collide（自碰撞）

    - 设置是否允许 link 之间发生自碰撞。没设时用 Gazebo 默认，可能导致腿间穿透或者过度碰撞

- 关节固定合并 `disableFixedJointLumping`

    - Gazebo 默认会把固定关节的 link 合并，合并后脚端 link 可能消失或接触检测异常

- 接触刚度/阻尼 kp / kd

    - 没设置时 Gazebo 接触模型偏“软”，脚可能下陷或抖动

### 姿态信息配置

在legged\_control中有两个姿态的概念，一个是初始姿态，一个是参考姿态或者默认姿态

- 初始姿态：`task.info`里的`initialState`：给 MPC 初始化用的“当前状态猜测”，应该尽量接近机器人启动时真实姿态（仿真里也一样）。可以是趴着，也可以是站立

- 参考姿态：`reference.info`里的`defaultJointState`\+`comHeight`：作为目标/参考姿态（例如 `cmd_vel/goal` 的名义姿态），通常设置成站立/中立姿态，但也可以改

如果你希望“初始是趴着，目标是站立”，就把`initialState`设成趴着、`defaultJointState`设成站立即可。如果你启动时就是站立，两个都设成站立即可

如果进一步扩展则有零点的概念，总体上大概有三层概念，运动学零点、机械/传感器零点、控制零点

- 运动学零点：模型里 q=0 的参考构型

    - 完全由 URDF 的关节轴 \+ 关节原点 \+ link 坐标系定义，DH 中体现为偏置角/偏置位移

    - 它决定了父子坐标系在 q=0 时的相对位姿，并且运动学零点不是“绝对零点”，而是你人为选择的“参考姿态”

    - OCS2/Pinocchio 的运动学计算默认就是这个零点

    - 一般**选一个对称、接近中位、数值稳定的姿态作为零点**（比如腿略微弯曲的站立姿态）

- 机械/传感器零点

    - 机械零点是装配或机械限位定义的物理零位，通常不等于模型零点

    - 传感器零点是编码器读数为 0 的位置，可能与机械零点一致，也可能不一致

- 控制零点

    - 控制器里定义“q=0”的基准，往往是机械零点 \+ 一个 offset映射到运动学零点后的结果

对零点和初始姿态的配置要求很高，一旦配置出现问题就会导致求解出错，从而导致仿真中机器人出现乱动/抖动的情况，无法正常运动，目前已经进行的尝试是，SolidWorks中已经定义好了运动学零点，在xacro中似乎无法修改真正意义上的运动学零点，因此这里直接使用导出的原始URDF中的关节状态作为零点，可以保证实现起来没有错误存在

为方便开发，本人自己制作了一个Dockerfile文件用于构建legged control项目，提供文件如下

\[Dockerfile\]

同时本人基于云原生方法，在国内服务器上构建了Docker镜像，大家可以直接自行拉取并运行，不必再自行构建，同时拉取速度十分有保障，拉取方法如下，同时提供镜像仓库地址：[legged control镜像仓库](https://cnb.cool/gpf2025/legged_control)，在该仓库的制品一栏中可以看到对应的镜像

```Shell
docker pull docker.cnb.cool/gpf2025/legged_control:v3
```

拉取下来之后就可以运行，关于docker的一些具体使用方法在网上有大量教程，因此在这里不在赘述，只提供一个运行指令并附详细解释

```Bash
docker run -it --rm \
    -e "DISPLAY=$DISPLAY" \
    -v "/tmp/.X11-unix:/tmp/.X11-unix:rw" \
    -e NVIDIA_DRIVER_CAPABILITIES=graphics,utility,compute,display \
    -v "$HOME/.Xauthority:/root/.Xauthority:rw" \
    -v "$(pwd)/outdoor.bag:/tmp/rosbag.bag" \
    --gpus all \
    -e "QT_X11_NO_MITSHM=1"  \
    --network=host  \
    docker.cnb.cool/gpf2025/legged_control:v3
```

- `-e "DISPLAY=$DISPLAY"`可视化用

- `-v "/tmp/.X11-unix:/tmp/.X11-unix:rw"`可视化用

- `-v "$HOME/.Xauthority:/root/.Xauthority:rw"`可视化用

- `-v "$(pwd)/outdoor.bag:/tmp/rosbag.bag"`将本地数据包挂载到镜像中使用

- `--gpus all`使用主机GPU

- `-e "QT_X11_NO_MITSHM=1"`可视化用

- `docker.cnb.cool/gpf2025/legged_control:v3`要运行的镜像名

因为Gazebo仿真器需要GPU支持才可以流畅运行，所以建议主机配有英伟达显卡并装好驱动，如果没有则会导致运行该镜像的Gazebo时非常卡顿

## 1. 双足点足实战 → **Ch B `legged_biped`**

**章名建议**：`从四足到双足点足：欠驱倒立摆、落脚平衡与硬约束 OCS2`
**label**：`ch:legged_biped`　**文件**：`parts/P7_legged/legged_biped.tex`
**放置**：`quad_prac` 之后（先讲完四足全栈+实践，再讲「移植到点足」最自然）。
**一句话定位**：四足靠静稳多边形「站得住」，点足是**主动不稳倒立摆**「只能靠迈步站住」——本章是「四足栈如何移植到欠驱点足」的方法论 + 公式底座 + Braver 实证。

### 节级大纲

**B.0 前置自测 / 本章目标 / 知识导航**（遵 v5.0 规范，仿 quad_wbc 体例）
- 自测题锚点：四足纯前馈 base 任务能站、点足为何不能？capture point 物理意义？为何点足「站立」必须是「踏步」？
- 承上：`\cref{ch:quad_arch,ch:quad_gait,ch:quad_mpc,ch:quad_wbc}`；前置桥接：`ch:quad_gait`（落足）、`ch:quad_mpc`（LIPM/质心）、`ch:quad_wbc`（base 任务/零空间）、`ch:lie`（旋转误差）。

**B.1 为什么点足「特别」：从静稳多边形到主动不稳倒立摆**
- 维度对照表（源 §1.3）：四足 stateDim=24/numContacts=4 vs Braver 点足 stateDim=18/inputDim=12/WBC=24/numContacts=2。
- 核心立论：两点接触在 ±0.1475m 横向但**前后同一 x 线** → pitch 方向无支撑力臂 → 支撑「多边形」退化成一条线 → **点足没有静止站立稳态**（源 §3.7「点足无静稳」）。
- `insight`：承 `ins:qa-floating-base`（浮动基欠驱）——四足欠驱但被 4 脚多边形「接住」，点足把这条姿态反馈通道掐断就发散。
- `pitfall`：把四足「base 反馈可选」直觉搬到点足（源 §3.2 致命根因的概念预告）。

**B.2 LIP / DCM / capture-point 完整推导**（源 §3.5 核心概念底座，接 `sec:srbd`/`sec:qg-footstep`）
- LIP 动力学 `ẍ=ω²x`（**正号→不稳定**），`ω=√(g/h)`，Braver `h≈0.427m→ω≈4.8 rad/s`。
- 发散模态分离 → **DCM（瞬时捕获点）** `ξ=x+ẋ/ω`，求导得 `ξ̇=ωξ`（一阶线性发散 `ξ(t)=ξ(0)e^{ωt}`）。
- **capture point** `p_capture=ξ=x+ẋ/ω`（令 `ξ−p=0`）；落 capture point 一步停稳、落偏后则残余 DCM 驱动下一步。
- `insight`：**为何纯 PD 稳 DCM 不迈步不行**——唯一抓手是穿过固定点接触的接触力（受摩擦锥、无力臂调 CoP），ξ 跑出脚附近 `e^{ωt}` 超出反馈修正 → **必须移动支撑点（迈步）重置 ξ**。这是点足「无静稳、横向平衡只能靠落脚」的数学根。
- **承上钩子**：明确指出 `eq:qg-raibert-vel` 里那个 `√(z0/g)(v−vdes)` **就是 capture-point 反馈项**——把 `quad_gait` 已写的工程式落足公式，在这里补上 DCM 理论解释（闭环 `\cref{sec:qg-footstep}`）。
- `remark`：本项目 MPC 不显式算此公式，但隐式优化落脚 XY 在数学上就是在做 capture-point-like 的事。

**B.3 ALIP 横向落脚闭式律**（源 §3.5，frontal plane 平衡的硬核公式）
- 横向平均速度为 0 却须左右交替以非零步宽 W 走 → period-2 极限环。
- 交替步宽闭式律 `L_x^des = ±½ mHW·ℓsinh(ℓT)/(1+cosh(ℓT))`（ℓ=横向 LIP 频率，T=步时）。
- `insight`：少了主动横向落脚反馈，**roll 模态必发散**（横向平衡只能靠落脚，呼应 Kuo/Bauby「人靠调步宽而非步长稳横向」）。
- capture-point/DCM 投影几何（反应式 capture 层用法）：`dcm=base_xy+world_vel/ω`、`projected_dcm=stance_xy+exp(clamp(ω·t,0,1.5))·(dcm−stance_xy)`（指数须 clamp）。
- 文献定位：ALIP（Gong&Grizzle）、ALIP-MPC（Gibson/Gong/Grizzle，落脚为决策变量）、H-LIP（Xiong&Ames）、DCM 步位+步时（Khadiv）。

**B.4 Braver OCS2 hard 约束接入**（源 §2.1★/§3.4 工程实战，接 `sec:qp`/`sec:state-space`）
- OCS2 OCP「五块拼图」+ ReferenceManager 设计因果链（源 §2.1★，**架构最重要心智模型**）：为并行→OCP 必须可拷贝→不能直接改 problem→需 ReferenceManager/SolverSynchronizedModule。三段式：ROS 回调加锁缓冲→`preSolverRun` 安全写入→求解器 `modifyReferences` 查询。
- Braver 真实配置（源 §2.1★，对双足才对的值）：`frictionCoefficient=0.6`（非四足 0.3）、`comHeight=0.42737`、`sqpIteration=3`、`timeHorizon=1.0s`、点足无静稳→`Q[6:12]=[1000,1000,2000,100,400,400]`（z=2000 守站高）。
- **OCS2 接入 hard 不等式清单**（源 §3.4）：继承 `StateInputConstraint`→声明 `ConstraintOrder`→实现 `isActive()/getNumConstraints()/getValue()/getLinearApproximation()`→OCP `inequalityConstraintPtr` 按脚注册。
- `pitfall`：**注册 hard 约束 ≠ 闭环满足**（源 §3.4 砸实数字：hard `Fz≥50N` 时 `inequalityConstraintsSSE=66680`、动态不一致反增）；命名陷阱 `frictionConeSoftConstraint.mu` 是惩罚权重不是摩擦系数。
- `pitfall`：软圆锥默认 `regularization=25` 且**不含 `Fz≥0` 项→负法向力对软锥「免费」**（解释 OCS2 desired 频繁负 stance Fz）。

**B.5 点足 WBC 与拆分 base 任务**（源 §3.7 突破，接 `sec:qw-floating`/`sec:qw-relax`）
- 修复（phase133）：`formulateBaseAccelTask = BaseXYLinearAccel(前馈) + BaseHeightMotion(Jacobian+PD) + BaseAngularMotion(Jacobian+旋转矩阵误差+世界系ω)`。
- 角任务正确形式：`a=base_j_(3:6)`，`b=accDes+Kp·rotationErrorInWorld(R_des,R_meas)+Kd·(ω_des−ω_meas)−base_dj_(3:6)·v`；期望量由 `computeBaseKinematicsFromCentroidalModel` **解析**得到（非有限差分）。
- `pitfall`：**旧前馈写法两块根因拼图**（源 §3.7）——(1) frame bug：ZYX-euler 角误差直接加到 body 系角加速度=三切空间相加、近 gimbal 奇异（正解须 `rotationErrorInWorld`+世界系 ω+base 角 Jacobian）；(2) 有限差分噪声放大：`jointAccel=(u−u_last)/period`，period≈0.0021s 放大约 476×——SRBD 下 Aⱼ≈0 幸免，**切 FullCentroidal 才致命**（解释 `centroidalModelType` 与 base 任务噪声耦合）。
- **承上钩子**：明确这是 `sec:qw-floating` 四子任务在「点足无静稳」下的特化——四足那套纯前馈 base task（quad_wbc 现有体例）对点足为何不够。

**B.6 运行时步态注入与「从一开始就走」**（源 §3.7，接 `sec:qg-scheduler`）
- 运行时注入 gait 有 ~1 horizon 延迟（`GaitReceiver` 在 finalTime 才 insert）→ 必须改 `reference.info` 的 `defaultModeSequenceTemplate`（DOUBLE_SUPPORT→standing_trot），首个 MPC policy 即规划踏步。
- `pitfall`：`ModeSequenceTemplate` 平铺陷阱（源 §3.5）——单模板 `[RIGHT_STANCE]` 被 `tileModeSequenceTemplate` 平铺成连续右支撑→左脚整 horizon swing；biped 一次性步需 `stance→DS→opposite stance→DS`；`switchingTimes` 必比 `modeSequence` 多一个（新手最常见崩溃源）。

**B.7 小结**：符号表 + 公式速查（LIP/DCM/capture/ALIP）+ 「四足栈→点足移植 checklist」+ 与 Ch C 失败方法论的指针。
- **失败方法论（层级归因谬误/规划 vs 执行二分）只在 B 章「点名预告」，完整论证放 Ch C**（避免与 Ch C 重复；B 章保持「方法+公式+实证」纯度）。

## 2. 感知运控实战 → **Ch A `legged_perceptive`**

**章名建议**：`感知四足运动控制：从盲 NMPC 到地形感知（OCS2 perceptive）`
**label**：`ch:legged_perceptive`　**文件**：`parts/P7_legged/legged_perceptive.tex`
**放置**：建议**在 Ch B 之前**（先把四足栈「升级到感知」，再讲「移植到点足」；二者独立，但感知是四足主线的直接延伸，认知负荷更低）。
> 备选顺序：A 在 B 后亦可（B 是「换机型」、A 是「加感知」，互不依赖）。推荐 A→B：保持「四足→四足+感知→双足」的能力递进。
**一句话定位**：盲 locomotion = 假设平地；感知 locomotion = 把地形（高程图→凸区域→SDF）作为**软约束**喂进 OCS2 NMPC，使机器人能在台阶/碎石上规划落脚。**这是 P7↔P2 SLAM 的接驳章**。

### 节级大纲

**A.0 前置自测 / 本章目标 / 知识导航**
- 承上：`\cref{ch:quad_mpc,ch:quad_wbc,ch:quad_gait,ch:quad_est}`；接 P2：`\cref{ch:pointcloud,ch:dense}`。
- 自测题：盲 MPC 假设了什么地形？地形怎么进 OCP（硬还是软约束）？为什么落脚点选区要按高度差排序、又为何这是个坑？

**A.1 感知 locomotion 范式：where（选落脚点）vs how（求力轨迹）**（源感知 §2.0/2.1）
- `insight`（心智模型）：**感知 NMPC =「启发式选带高度的落脚点 where」→「质心动力学 NMPC 求力/轨迹 how」→「WBC 解成关节力矩」**。地形只通过两条途径进入：①`ConvexRegionSelector` 投影落脚点到凸区域；②`ReferenceManager::modifyReferences` 抬 base-z + 给 base-pitch。**无显式 ZMP/支撑多边形/roll 约束**，稳定全靠落脚点+摩擦锥+地形法向 pitch 隐式保证。
- 类继承（源感知 §1.2）：`PerceptiveLeggedInterface:LeggedInterface`、`PerceptiveController:LeggedController`——**WBC/Gazebo/KF 全复用、零改动**，感知只在 NMPC cost/constraint 层叠加 + 换 ReferenceManager。承 `quad_wbc`：说明为何 WBC 一行不用改。

**A.2 感知数据流与 elevation mapping**（源感知 §1.3/2.x，**接 P2 SLAM 核心**）
- 完整链（数据流图）：`深度相机/lidar 点云 → robot_self_filter → elevation_mapping_cupy(2.5D 机体跟随高程图) → convex_plane_decomposition(凸平面区域+smooth_planar) → PlanarTerrainReceiver(SynchronizedModule) → ConvexRegionSelector + PerceptiveReferenceManager → NMPC(FootPlacement+FootCollision) → WeightedWbc`。
- **接 `\cref{ch:dense}`**：elevation map = 2.5D 稠密建图，与 OctoMap(`sec:dense-octomap`)/TSDF(`sec:dense-tsdf`) 同族但**机体跟随、单层高度**；`insight` 对比「全局 3D 占据 vs 机体局部 2.5D 高程」的取舍。
- `SynchronizedModule` 机制 = OCS2 每 MPC 周期前线程安全把外部地形喂进求解器（呼应 B.4 的 ReferenceManager 因果链）。

**A.3 self-filter / plane segmentation / SDF**（**接 P2 SLAM `ch:pointcloud`/`ch:dense`**）
- **robot_self_filter**：滤掉机器人自身点云 → **接 `\cref{sec:pc-filter}`（点云滤波，self-filter ≈ 几何剔除滤波，与 SOR/ROR/体素并列）**。
- **convex_plane_decomposition**：高程图分割成凸平面区域 → **接 `\cref{sec:pc-paradigm}`（点云分割范式）**；多边形→半空间精确构造（源 §2.4）：`a_ℓ=[y_{ℓ+1}−y_ℓ; x_ℓ−x_{ℓ+1}]`、`b_ℓ=y_ℓx_{ℓ+1}−x_ℓy_{ℓ+1}`，下一顶点判内法向翻转。
- **Signed Distance Field（足-地形碰撞）** → **接 `\cref{sec:dense-tsdf}`（TSDF/ESDF）+ voxblox-ESDF**；SDF 层 `elevation_before_postprocess`，分辨率 0.03，范围 ±0.1。

**A.4 落脚点选择 ConvexRegionSelector + dz² 坑**（源感知 §2.2/§3.4，最易踩坑的环）
- 名义落脚 `getNominalFoothold`：`foothold=FK(desiredState)[leg]−R^T·offsetVector`，`offsetVector=tan(−pitch)·0.4`（base-pitch 把落点往坡上推=爬升机制）。
- 投影到「最优」凸区域：`cost=distanceCost(含 dz²)+penaltyFunction`。
- ⭐`pitfall`（**dz² 架构缺陷，源 §3.4，多 Agent 收束最佳诊断**）：名义落点 z≈0 时地面区 dz²=0 赢、台阶顶区 dz²=0.0256 输 → **偏选地面区→脚留地面**；base 摇晃→赢家 step↔ground 逐周期闪烁→对角 trot 撑不住→横滑翻；swing-apex 饥饿（touchdown z≈0→apex 0.08<0.16→撞 riser）；**鸡生蛋**（脚上台阶要 base 先抬，base 抬要脚先上）。修法：no-op penalty 换成「区平面低于脚下局部地形高」的惩罚 `w·max(0,terrainZ−p.z())²`。

**A.5 Perceptive NMPC 约束与 base 参考构造**（源感知 §2.1/2.3，接 `sec:qp`/`sec:state-space`）
- 两条 per-foot 软约束（`RelaxedBarrierPenalty`，非硬约束以防不可行）：**FootPlacementConstraint**（落在凸多边形内）+ **FootCollisionConstraint**（脚 vs 地形 SDF，clearance=0.03，仅 swing 时 active）。
- base 参考三通道（源感知 §2.3）：base-PITCH（4 足平面拟合 dzdx→`atan`，clamp ±0.12rad）、base-Z（`zBase+comHeight/cos(pitch)` 单点）、base-ROLL（默认 0；坡上 banking=平面侧向梯度 §2.5）。
- `pitfall`：**`positionErrorGain=0` 摆腿 z 开环**（源感知 §2.3，与 dz² 同级真缺陷）——上游默认 20、港口三机继承 0 → 跳过摆腿 z 位置反馈→撞竖板/漂走无纠正。
- `insight`：**强 base-z 参考是非规范简化**（源感知 §2.3）——文献（Grandia/Corbères/DTC）倾向「隐藏 base 参考」，靠落脚点+摆动爬台阶；这解释 over-launch 触发层（详见 Ch C）。

**A.6 感知 WBC（六任务）与纯力矩下发**（源感知 §2.4，接 `sec:qw-floating`）
- 六任务表：FloatingBaseEom/TorqueLimits/FrictionCone/NoContactMotion/BaseAccel/SwingLeg。
- `pitfall`：**权重进 H=AᵀA 是平方**（源感知 §2.4）——`swingLeg:baseAccel=100²:1=10000:1` 摆腿碾压 base-roll 恢复。
- 底层下发 `kp=0,kd=3,ff=τ`（纯力矩+小阻尼，所有机器人共享→「调底层 PD」无机型差异）。
- **承上**：这套 WBC 与 `quad_wbc` 的 WeightedWbc 同源，本节只补「感知不改 WBC」+ 平方权重坑。

**A.7 接 P2：感知接触检测（gait-clock vs 实测）**（源感知 §4，接 `sec:qe-contact`）
- `insight`：**WBC/调度用 gait-clock 接触而非实测**（grep `contactDetection`=空）→ 台阶处前脚晚着地、调度已置 STANCE→质心模型假设不存在的 GRF→roll。**接 `\cref{ins:qe-prob-boundary}`（平地够用/复杂地形必须检测）+ `sec:qe-gm`（GM 观测器）+ `bledt2018contact`（event-based locomotion）**。

**A.8 小结**：符号表 + 感知数据流速查 + 「移植新机器人到感知框架 checklist」+ 失败根因（关节阻尼/over-launch）指针到 Ch C。
- **关节阻尼/over-launch 失败根因只点名预告，完整证伪链放 Ch C。**

### 节级大纲

**C.0 前置 / 目标 / 导航**
- 承上：`\cref{ch:legged_perceptive,ch:legged_biped}`（两章的失败根因在此完整展开）。

**C.1 三层可行性诊断**（源双足 §3.4★「升级为通用三层可行性诊断」，本章方法论支柱）
- ① 当前 tick WBC QP `lastSolveSucceeded()`（此刻解出来了吗）；② MPC 解本身 `getPerformanceIndices()` 六字段（`dynamicsViolationSSE/equalityConstraintsSSE/inequalityConstraintsSSE/cost/merit/Lagrangian`）；③ policy 预测窗口 `evaluatePolicy(t+0.3/+0.6)` 看 pitch/height 是否单调发散（计划里就要摔）。
- ⭐`insight`：**最隐蔽失败模式**——三层标量都不报警（WBC 每 tick 解成功 + MPC 用质心模型不知 WBC 把姿态反馈调弱了 + 缓慢倒），**必须靠多维指标（roll/pitch 缓慢漂移）+ repeat 方差才能抓**。这是「增益被悄悄置零为何 ~120 phase 难定位」的诊断学解释。
- 概念纠错：金字塔 vs 圆锥 feasible-set 不一致（OCS2 软圆锥度量 vs WBC 线性金字塔残差）；`sqp.g_max` 收敛≠硬不等式满足（唯一证据是 `inequalityConstraintsSSE`）。

**C.2 层级归因谬误与「规划 vs 执行」二分**（源双足 §3.2★，全套文档最高价值）
- 病根：根因在 WBC 执行层（base 反馈增益恒为 0），却 ~120 phase 坚信瓶颈在 OCS2 上游（落脚/步时/力可行性）。
- 四种归因谬误：层级归因谬误 / 四足直觉误导 / 症状掩盖 / 创可贴陷阱。
- ⭐**最强单一定位杠杆**：每拍同时打印 **controller 实测量 与 MPC 对未来的预测量**——一旦对照立刻证明 MPC 规划全程对的（完美 48N/脚 站立解、预测 pitch→0），实测却 50ms 内 pitch 0.004→−0.25 比倒立摆自然发散快 17 倍=控制器**主动推反**→问题 100% 在执行层。
- 可复算基线：LIP `ω₀=√(g/h)≈4.67~4.8 rad/s`，自然倒下需 ~0.9s；实测快约 17 倍=量化证明「主动推反」。
- `pitfall`：「力矩饱和 ≠ 力矩不足」（源双足 §3.4）——单支撑横向实际只需约 14Nm、限幅 40Nm=2.8× 余量→饱和是被失控倒立摆逼到的后果。

**C.3 over-launch 证伪链（感知侧案例研究）**（源感知 §3.3★★★，教科书级「错误转弯比正确结论更有教学价值」）
- 现象：dmgo 上 0.16 台阶竖直冲到 z=0.48–0.68 + 前冲 + 随机方向 + 翻 roll±180；QP 干净；方向随机（关键线索）。
- **13 行假设证伪表**（逐个提出又被实验证伪，源感知 §3.3）：感知覆盖/swing 激进/**2.27× roll 惯量(更轻翻更狠=amplifier 非 gate)**/CoM 高/关节阻尼/CppAD 缓存(红鲱鱼)/gait-clock 接触/力权 Q/base-z 不一致/base-ori 软任务/p_base_z 软化/非飞行步态/速度调度(过驱动非欠驱)。
- ⭐`pitfall`：**升 roll 权重是错方向**（源感知 §3.3 表行 10，P3-082）——OCS2 代价是质量+惯量归一化 `Q(3,3)·(I_xx·ω/m)²`，2.27× roll 惯量已让同权重自动放大 2.27×→应降到 ×0.44 而非升 ×4。可迁移到任何质心 MPC 调权重。
- 文献交叉验证（源感知 §3.3，arXiv 2605.05707 的 Thm 1/5/6/7）：N=2 stance 摩擦锥对 ‖Ḣ_G‖ 有几何下界「independent of weight tuning」；临界水平加速度 `a_x⋆=μg/(1+2μκ)`；N≥3 降 ‖Ḣ_G‖~227×；**注入角动量参考 FUTILE**（Thm 7）。over-launch=飞行相 CAM 失稳（arXiv 2501.17351）。
- 诚实定论：maxz 双峰（0.31 卡/0.46 弹，无 0.38 渐爬）=重躯干内禀属性（但「方法的墙非物理的墙」——HyQ 80kg 爬 24cm 靠 centroidal 而非位置跟踪）。

**C.4 统计功效铁律与 setup≠result**（源感知 §3.3b/§3.7，统计学根源）
- ⭐`insight`：**N=4 区分 30%↔10% 需 40-85 trial=个位数功效**→本项目所有 N≤8 结论（μ0.6「1/4 首过」、各「0/4 证伪」）统计上无效=反复翻车循环真正根源。
- `pitfall`：**setup≠result**（源感知 §3.7 第1条）——记忆/总结的「铁证」必须回溯原始 `debug_log.json` 的 `result` 字段；doc29 把「换惯量测试」(setup) 误读成 result(实为 REFUTED)。
- `pitfall`：**单次优势必须 ≥3 repeat A/B 才能成默认**（源双足 §3.5）——身体倒下初始 roll/pitch 是随机主导项，单次「长 run」是 early-state 方差非控制收益。

**C.5 证伪驱动方法论全集（12 条横切教训）**（源感知 §3.7，最可迁移）
- 列举（每条配一句源项目实证）：别死磕调参(三-strikes)/proxy≠truth/common-mode 假收敛(查独立性)/证伪优先/直接测量胜推断/EXTREME-BRACKET(区间夹逼)/CLIFF-DISCRIMINATOR(别在饱和区测)/多试次 pass-rate/基线先行(measured≠planned)/广度先于深度/Agent 研究必交叉验证失败史。
- `insight`：「纪律即生产力」单会话 3 连救场实证。

**C.6 静默坑与工程基建合集**（源感知 §3.1/3.2/3.6/3.8，可复用工程教训）
- friction-cone μ 失配（**controller μ=½ sim μ 安全余量律**，源感知 §3.1 最重要真修，可迁移为「为何 MPC 摩擦系数要比真实小」范例）。

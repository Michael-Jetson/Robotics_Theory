# §2 引子：小萝卜的例子（源 §2.1）

## §2.1 自主运动 → 需要定位与建图

源以一台自制机器人「小萝卜」（设备含相机、轮子、笔记本电脑塞在后备箱，手为装饰）引入。核心论证链（逐点保真）：

- 我们希望机器人具有**自主运动能力**——这是许多高级功能（扫地、搬东西）的前提。要移动需轮子和电机；但不加规划与控制，机器人会乱走甚至撞墙。
- 要规划与控制，**首先需感知周边环境**——故装相机（动机：机器人与人相似，靠"眼睛+大脑+四肢"在任意环境行走探索）。
- 为探索一个房间，机器人**至少需知道两件事**：
  1. **我在什么地方？——定位（Localization）。**
  2. **周围环境是什么样？——建图（Mapping）。**

**"内外之分"的提法（源原话）**：「"定位"和"建图"，可以看成感知的"内外之分"。作为一个"内外兼修"的小萝卜，一方面要明白自身的状态（即位置），另一方面也要了解外在的环境（即地图）。」

> **★ 定位与建图的相互依赖（"鸡生蛋"问题，SLAM 的本质动机）**：源在此处尚未明写"鸡生蛋"四字，但全章逻辑即此——定位需要地图作参照，建图又需要已知位姿才能把观测放到正确位置；二者互为前提、必须**联合估计**。这正是 SLAM（Simultaneous Localization and Mapping，同时定位与建图）的名字由来。综合写正文时应在本节点明这一相互依赖。

## §2.2 传感器的两大分类（源 §2.1）

源把传感器（图 2-2）分为两类：

| 类别 | 定义 | 典型设备 | 优点 | 缺点/约束 |
|---|---|---|---|---|
| **安装于环境中** | 部署在环境里的传感设备 | 导轨/导引线、二维码标志、无线电定位设备、（室外）GPS 接收器 | 通常**直接测量机器人位置**，简单有效解决定位 | **要求环境必须人工布置**，限制使用范围（室内无 GPS、园区难铺导轨） |
| **携带于机器人本体** | 装在机器人自身的传感器 | 轮式编码器、相机、激光传感器、IMU（Inertial Measurement Unit，惯性测量单元） | **不对环境提出任何要求**，适用于未知环境 | 测量的是**间接物理量**（轮编→轮子转角；IMU→角速度/加速度；相机/激光→外部环境观测），只能**间接推算**位置（"迂回战术"） |

**图 2-2 子图清单（图注逐字）**：(a) 利用二维码进行定位的增强现实软件；(b) GPS 定位装置；(c) 铺设导轨的小车；(d) 激光雷达；(e) IMU 单元；(f) 双目相机。

**关键结论（源原话精炼）**：
- 环境类传感器「约束了外部环境……当约束无法满足时，我们就无法进行定位」，故"简单可靠，但无法提供普遍、通用的解决方案"。
- 本体类传感器"没有对环境提出任何要求，从而使得这种定位方案可适用于未知环境"。
- **SLAM 强调未知环境**：「我们在 SLAM 中非常强调未知环境。所以理论上，我们不限制小萝卜的使用环境……因此，使用携带式的传感器来完成 SLAM 是我们重点关心的问题。」
- **视觉 SLAM 定义**：「当谈论视觉 SLAM 时，我们主要是指如何用相机解决定位和建图问题。」对应地，主用激光的称**激光 SLAM**（相对成熟；2005《概率机器人》[5] 已介绍许多激光 SLAM 知识；ROS 里有现成激光定位/建图软件）。

**源脚注（保真）**：① 本体传感器"没有对环境提出任何要求"。② 不限制小萝卜的使用环境（以上两脚注在源中标记于该段，强调通用性）。

## §2.3 相机分类（源 §2.1）

SLAM 中的相机不同于单反摄像头：更简单、通常不带昂贵镜头，以一定速率拍摄形成**连续视频流**。普通摄像头约 **30 帧/秒**，高速相机更快。按工作方式分三大类（图 2-3）+ 特殊类：

| 相机类型 | 英文 | 摄像头数 | 能否直接测深度 | 原理/特点 | 讲解所在 |
|---|---|---|---|---|---|
| 单目 | Monocular | 1 | 否 | 二维记录三维，丢失深度维 | 本节 |
| 双目 | Stereo | 2 | 能（软件计算） | 已知基线 Baseline，比较左右眼差异估深度 | 本节 + 第5讲 |
| 深度 | RGB-D | 多个（含主动发射） | 能（物理测量） | 红外结构光或 ToF | 第5讲详述 |
| 全景 | Omnidirectional/Panoramic | — | — | 特殊种类，SLAM 中偶见，非主流[7] | — |
| 事件 | Event camera | — | — | 新兴种类，非主流[8] | — |

> 源原话：「从样子上看，小萝卜使用的似乎是水平的双目相机。」

### §2.3.1 单目相机（源「单目相机」小节）

- 只用一个摄像头的 SLAM = **单目 SLAM（Monocular SLAM）**。结构简单、成本低、受研究者关注。
- 单目数据 = 照片 = "某个场景（Scene）在相机成像平面上留下的一个投影"，**以二维记录三维，丢掉了深度（距离）维**。
- **无法从单张图片算出物体与相机的距离**。人靠经验/直觉判断远近（物体大小常识、近处遮挡远处、天体很远、光照阴影），但这些会失效（图 2-4：手掌上的小人是真人还是模型？无法仅凭单图判断）。
- **结论**：单张图像里无法确定物体真实大小——"很大但很远" 与 "很近但很小" 因**近大远小的透视关系**在图像中可能等大。
- **要恢复三维必须改变视角 / 移动相机**：移动相机才能估计它的**运动（Motion）**，同时估计场景物体远近大小（称**结构 Structure**）。
- **视差（Disparity / Parallax）**：相机移动时，近处物体移动快、远处慢、极远（无穷远，如日月）看上去不动；由此形成视差，"通过视差，我们就能定量地判断哪些物体离得远，哪些物体离得近"。
- **尺度不确定性（Scale Ambiguity）**：即使知道相对远近，仍只是相对值。把相机运动与场景大小同时放大任意倍，单目所见一样。故 **单目 SLAM 估计的轨迹和地图与真实相差一个因子——尺度（Scale）**；单目无法仅凭图像确定真实尺度，称尺度不确定性。
- **单目两大根本麻烦**（源原话）：「平移之后才能计算深度，以及无法确定真实尺度」，根本原因是**通过单张图像无法确定深度**。→ 为得到深度，引入双目相机与深度相机。

**源脚注**：① 尺度（Scale）——单目重建与真值相差的整体比例因子。

### §2.3.2 双目相机与深度相机（源「双目相机和深度相机」小节）

**双目相机（Stereo）**：
- 由两个单目相机组成，二者间距 = **基线（Baseline）**（已知）。通过基线估计每个像素的空间位置——类比人眼通过左右眼图像差异判断远近（图 2-5）。
- 可拓展为**多目相机**（multi-camera），本质无不同。
- 计算机双目需**大量计算**才能（不太可靠地）估计每个像素深度，比人类笨拙。
- **测量深度范围与基线相关**：基线越大可测越远 → 无人车上的双目相机通常很大。
- 既可室内又可室外（不依赖其他传感设备，仅比较左右图）。
- **缺点**：配置与标定复杂；深度量程与精度受基线和分辨率限制；视差计算非常耗算力，需 GPU/FPGA 加速才能实时输出整张图像距离。**计算量是双目主要问题之一。**

**深度相机（RGB-D，本书统一称 RGB-D）**：
- 2010 年前后兴起。最大特点：通过**红外结构光**或 **Time-of-Flight（ToF）** 原理，像激光那样**主动发射光并接收返回光**，物理测出距离。
- 相比双目（软件计算）可**节省大量算力**（图 2-6）。
- 常用型号：**Kinect / Kinect V2、Xtion Pro Live、RealSense** 等；部分手机用于识别人脸。
- **现存问题**：测量范围窄、噪声大、视野小、易受日光干扰、无法测量透射材质等。SLAM 中**主要用于室内，室外较难应用**。

**图注**：图 2-5 双目相机数据（左眼/右眼图像，靠差异判断距离）；图 2-6 RGB-D 数据（直接测物体图像与距离，恢复三维结构）。

**承上启下（源原话）**：「想象相机在场景中运动……得到一系列连续变化的图像。视觉 SLAM 的目标，是通过这样的一些图像，进行定位和地图构建……SLAM 需要一个完善的算法框架，而经过研究者们长期的努力工作，我们已经发展出了一套比较成熟的框架。」

**源脚注**：① 计算机双目"相比人类真是非常笨拙"。② 相机运动得到"一系列连续变化的图像"。

# §3 经典视觉 SLAM 框架（源 §2.2）★ 本章核心

## §3.1 五模块总览与数据流（源 §2.2 + 图 2-7）

源给出**经典视觉 SLAM 框架（图 2-7）**，整个流程包括以下步骤（逐字保真，编号即源编号）：

1. **传感器信息读取（Sensor data）**。视觉 SLAM 中主要为相机图像信息的读取和预处理。机器人中还可能有码盘、惯性传感器等信息的读取和**同步**。
2. **前端视觉里程计（Visual Odometry, VO）**。任务：估算相邻图像间相机的运动，以及局部地图的样子。VO 又称**前端（Front End）**。
3. **后端（非线性）优化（Optimization）**。后端接受不同时刻 VO 测量的相机位姿，以及回环检测的信息，对它们进行优化，得到**全局一致的轨迹和地图**。因接在 VO 之后，又称**后端（Back End）**。
4. **回环检测（Loop Closure Detection）**。判断机器人是否到达过先前的位置。若检测到回环，把信息提供给后端处理。
5. **建图（Mapping）**。根据估计的轨迹，建立与任务要求对应的地图。

**框架图（图 2-7 数据流，本抽取按文字重建为 ASCII，供综合作图）**：

```
   ┌────────────┐
   │ 传感器数据读取 │  相机图像（+码盘/IMU），预处理与同步
   │  (Sensor)   │
   └─────┬──────┘
         │ 图像/帧
         v
   ┌────────────┐
   │ 前端视觉里程计 │  相邻帧→相机运动估计 + 局部地图
   │  VO/Front  │
   └─────┬──────┘
         │ 相机位姿（带累积漂移）+ 特征/局部地图
         v
   ┌────────────┐      ┌──────────────┐
   │  后端优化   │<────│  回环检测      │  "A 与 B 是同一点"约束
   │ Back End/   │ 约束 │ Loop Closure  │  (判断是否回到旧地点)
   │ Optimization│      └──────────────┘
   └─────┬──────┘
         │ 全局一致的轨迹 + 优化后的地图
         v
   ┌────────────┐
   │   建图      │  按任务需求生成地图（度量/拓扑/稠密/稀疏…）
   │  Mapping    │
   └────────────┘
```

**各模块输入/输出/职责对照表（本抽取整理，依据源 §2.2.1–§2.2.4）**：

| 模块 | 输入 | 输出 | 核心职责 | 又称 | 关键问题 | 详述章 |
|---|---|---|---|---|---|---|
| 传感器读取 | 相机图像（+码盘/IMU 原始数据） | 同步/预处理后的帧数据 | 读取与同步多源数据 | — | 多传感器时间同步 | 第5讲（相机模型）等 |
| 视觉里程计 VO | 相邻帧图像 | 相邻帧相机运动 + 局部地图（空间结构） | 估相邻图像间相机运动、恢复场景空间结构 | 前端 Front End | **累积漂移 Accumulating Drift** | 第7、8讲 |
| 后端优化 | 各时刻 VO 位姿 + 回环约束 | 全局一致轨迹 + 地图 + 不确定性 | 从带噪数据估计系统状态及其不确定性（MAP） | 后端 Back End | 噪声处理、状态估计 | 第6、9、10讲 |
| 回环检测 | 图像（当前 vs 历史） | "A 与 B 是同一点"的约束 | 判断是否回到旧地点，提供约束给后端 | 闭环检测 | 图像相似性计算 | 第11讲 |
| 建图 | 估计的轨迹（+ 观测） | 任务相应的地图 | 按应用构建地图 | — | 无固定形式/算法 | 第12、13讲 |

**框架成熟度结论（源原话）**：「经典的视觉 SLAM 框架是过去十几年的研究成果……如果把工作环境限定在**静态、刚体、光照变化不明显、没有人为干扰**的场景，那么这种场景下的 SLAM 技术已经相当成熟[9]。」

## §3.2 视觉里程计 VO（源 §2.2.1）

- VO 关心**相邻图像之间的相机运动**，最简单是两张图像之间的运动关系（图 2-8：右图是左图向左旋转一定角度的结果）。
- 人能直觉判断"向左旋转"（近处吧台、远处墙壁黑板；相机左转时近处吧台进入视野、右侧远处柜子移出），但**说不清旋转多少度、平移多少厘米**——直觉对具体数字不敏感。计算机却必须**精确测量**这段运动。
- 图像在计算机里只是**数值矩阵**，矩阵表达什么计算机毫无概念。视觉 SLAM 中只能看到一个个像素，知道它们是"某些空间点在相机成像平面上投影的结果"。**为定量估计相机运动，必须先了解相机与空间点的几何关系。**
- **"里程计"得名**：和实际里程计一样，只计算相邻时刻运动，与过去信息无关——像"只有短时记忆的物种"（不限于两帧，可 5～10 帧）。
- **有了 VO 能否解决 SLAM？** 一方面把相邻时刻运动"串"起来得轨迹（解决定位）；另一方面据各时刻相机位置算各像素对应空间点位置得地图。但——
- **累积漂移（Accumulating Drift）/ 漂移（Drift）**：VO 每次估计都有误差，先前误差传到下一时刻；经一段时间后轨迹不再准确（图 2-9）。
  - **数值例（源原话）**：机器人先向左转 $90^\circ$，再向右转 $90^\circ$。由于误差把第一个 $90^\circ$ 估成了 $89^\circ$，则右转后估计位置**回不到原点**；更糟：即使之后估计完全准确，与真值相比都会带上这 $-1^\circ$ 的误差。
  - 后果：无法建立一致地图——"原本直的走廊变成了斜的，而原本 $90^\circ$ 的直角不再是 $90^\circ$"。
- **解决漂移需两种技术**：**后端优化** 与 **回环检测**。回环检测负责把"机器人回到原始位置"检测出来；后端优化据此**校正整个轨迹的形状**。

**图注**：图 2-8 相机拍摄图片与人眼反应的运动方向；图 2-9 累积误差与回环检测的校正结果[10]。

**源脚注**：① 人"往往难以用理性的语言描述我们的直觉"。① 后端优化（该段标记，指向其与回环检测共同纠漂移）。

## §3.3 后端优化（源 §2.2.2）

- **后端优化主要指处理 SLAM 过程中的噪声问题**：再精确的传感器也带噪声（便宜的误差大、贵的小、有的受磁场温度影响）。
- 除了"如何从图像估计相机运动"，还要关心：这个估计带多大噪声、噪声如何从上一时刻传到下一时刻、我们对当前估计有多大自信。
- **后端优化的问题定义（源原话）**：「如何从这些带有噪声的数据中估计整个系统的状态，以及这个状态估计的不确定性有多大——这称为**最大后验概率估计（Maximum-a-Posteriori, MAP）**。这里的状态既包括机器人自身的轨迹，也包含地图。」
- **前端 vs 后端分工**：前端给后端提供**待优化的数据及其初始值**；后端负责整体优化，**面对的只有数据，不必关心来自什么传感器**。视觉 SLAM 中，前端更接近计算机视觉（图像特征提取与匹配），后端主要是**滤波与非线性优化算法**。
- **历史**：现在称"后端优化"的部分，很长一段时间内直接被称为"SLAM 研究"。早期 SLAM 是一个**状态估计问题**（正是后端要解决的）。最早提出 SLAM 的系列论文称它为**"空间状态不确定性的估计"（Spatial Uncertainty）[4,11]**——反映 SLAM 本质：「对运动主体自身和周围环境**空间不确定性**的估计」。
- **方法论**：需要状态估计理论把定位与建图的不确定性表达出来，然后用**滤波器或非线性优化**，估计状态的**均值和不确定性（方差）**。具体内容在第6、9、10讲。

## §3.4 回环检测（源 §2.2.3）

- **回环检测，又称闭环检测**，主要解决**位置估计随时间漂移**的问题。
- 机理：机器人运动一段时间后实际回到原点，但因漂移其位置估计没回到原点；若有手段让机器人知道"回到了原点"或把"原点"识别出来，再把位置估计"拉"过去，即可消除漂移。
- 回环检测与"定位"和"建图"都密切相关。**地图存在的主要意义是让机器人知晓自己到过的地方**；故需让机器人具有**识别到过场景**的能力。
- 实现手段：可设环境标志物（如二维码）——但那是环境中的传感器、对应用环境作了限制（不能贴二维码怎么办）。**更希望用携带的传感器——图像本身**：例如**判断图像间的相似性**完成回环检测（类比人看到两张相似图片易辨认来自同一地方）。
- **结论（源原话）**：「视觉回环检测实质上是一种**计算图像数据相似性**的算法。由于图像的信息非常丰富，使得正确检测回环的难度降低了不少。」
- 检测到回环后，把"**A 与 B 是同一个点**"的信息告诉后端优化；后端据新信息把轨迹和地图调整到符合回环结果。**有充分且正确的回环检测 → 消除累积误差，得到全局一致的轨迹和地图。**

## §3.5 建图（源 §2.2.4）

- 建图 = 构建地图的过程。**地图是对环境的描述，但描述并不固定，需视 SLAM 应用而定。**
- **图 2-10 地图种类（图注逐字）**：2D 栅格地图、2D 拓扑地图、3D 点云地图、3D 网格地图[12]。
- 应用差异（源举例）：
  - 家用扫地机器人（低矮平面运动）只需**二维地图**标记可通过/障碍物。
  - 相机有**6 自由度**运动 → 至少需**三维地图**。
  - 有时想要漂亮重建（不仅空间点，还要带纹理的三角面片）；有时不关心样子，只需"A 到 B 可通过、B 到 C 不行"；甚至不需要地图，或地图由他人提供（如行驶车辆得到已绘制的当地地图）。
- **结论**：相比 VO/后端/回环，**建图没有固定的形式和算法**；"一组空间点集合、一个漂亮 3D 模型、一张标记城市村庄铁路河道的图片"都可称地图。大体分**度量地图**与**拓扑地图**两种。

### §3.5.1 度量地图（Metric Map）

- 强调**精确表示物体位置关系**，通常用**稀疏（Sparse）与稠密（Dense）**分类。

| 度量地图子类 | 定义 | 表示 | 适用 | 代价/问题 |
|---|---|---|---|---|
| 稀疏（Sparse） | 一定程度抽象，不表达所有物体；只选有代表意义的=**路标（Landmark）**，非路标部分忽略 | 由路标组成 | **定位**用稀疏路标地图就够 | — |
| 稠密（Dense） | 着重建模**所有看到的东西** | 按某分辨率由许多小块组成：二维=许多**小格子（Grid）**；三维=许多**小方块（Voxel）** | **导航**往往需要（否则会撞上两路标间的墙） | 需存每个格点状态→**耗大量存储**，多数细节无用；大规模时**一致性问题**（小转向误差致两屋墙重叠，地图失效） |

- 稠密小块通常含**占据、空闲、未知**三种状态，表达该格内是否有物体；查询某空间位置时给出能否通过。
- 可用于各种导航算法，如 **A\*、D\*** 等，为机器人研究者所重视。

**源脚注**：① A\*、D\* 等导航/路径搜索算法（指向规划章）。

### §3.5.2 拓扑地图（Topological Map）

- 相比度量地图的精确性，**拓扑地图更强调地图元素之间的关系**。
- 拓扑地图是一个**图（Graph），由节点和边组成**，只考虑节点间的**连通性**（例如只关注 A、B 连通，不考虑如何从 A 到 B）。
- 放松了对精确位置的需要，去掉细节，是**更紧凑的表达**。
- **缺点/开放问题**：不擅长表达复杂结构的地图；如何对地图分割形成节点与边、如何用拓扑地图导航与路径规划，仍有待研究。

# §4 SLAM 问题的数学表述（源 §2.3）★ 核心数学

## §4.1 离散时间、状态量与路标量（源 §2.3）

把"小萝卜携带传感器在未知环境运动"上升到数学语言：

- 相机在**某些时刻**采集数据，故只关心这些时刻的位置与地图 → 把**连续时间运动离散化**为时刻 $t=1,\cdots,K$。
- 用 $\boldsymbol x$ 表示小萝卜自身位置；各时刻位置记为
$$
\boldsymbol x_{1},\ \boldsymbol x_{2},\ \cdots,\ \boldsymbol x_{K},
$$
它们构成小萝卜的**轨迹**。
- 假设地图由许多**路标（landmark）**组成，每时刻传感器测到一部分路标点，得其观测数据。设路标共 $N$ 个：
$$
\boldsymbol y_{1},\ \boldsymbol y_{2},\ \cdots,\ \boldsymbol y_{N}.
$$

"小萝卜携带着传感器在环境中运动"由两件事描述：

1. **什么是运动？** 考察从 $k-1$ 时刻到 $k$ 时刻，位置 $\boldsymbol x$ 如何变化。
2. **什么是观测？** 假设小萝卜在 $k$ 时刻于 $\boldsymbol x_k$ 处探测到某路标 $\boldsymbol y_j$，如何用数学语言描述。

## §4.2 运动方程（源 §2.3，式 2.1）

机器人携带测量自身运动的传感器（码盘/惯性传感器），读数不一定直接是位置之差，可能是加速度、角速度等；或人给指令（"前进1米""左转90°""油门踩到底""刹车"）。统一用抽象数学模型：

$$
\boldsymbol{x}_{k} = f\!\left(\boldsymbol{x}_{k-1},\, \boldsymbol{u}_{k},\, \boldsymbol{w}_{k}\right). \tag{2.1}
$$

- $\boldsymbol u_k$：运动传感器的读数或输入。
- $\boldsymbol w_k$：该过程中加入的**噪声**。
- $f$：一般函数，**不指明具体作用方式**，使方程可指代任意运动传感器/输入，成为通用方程。称为**运动方程（motion equation）**。

**随机性说明（源原话）**：「噪声的存在使得这个模型变成了**随机模型**。换句话说，即使我们下达'前进1米'的命令，也不代表小萝卜真的前进了1米。」小萝卜可能某次前进 0.9 米、另一次 1.1 米、再一次因轮胎打滑没前进。每次运动过程的噪声是随机的；若不理会噪声，只按指令确定的位置可能与实际相差十万八千里。

> **本书统一约定补注**：综合写正文时按本书约定补 $\boldsymbol w_k\sim\mathcal N(0,\Sigma_w)$（零均值高斯，协方差 $\Sigma_w$）。本章原文未给噪声分布，分布假设留到 §4.5 的"线性/非线性、高斯/非高斯"分类处讨论。

**源脚注**：① $\boldsymbol u_k$ 是"运动传感器的读数或者输入"。

## §4.3 观测方程（源 §2.3，式 2.2）

与运动方程相对应，描述小萝卜在 $\boldsymbol x_k$ 看到路标点 $\boldsymbol y_j$ 时产生观测数据 $\boldsymbol z_{k,j}$：

$$
\boldsymbol{z}_{k,j} = h\!\left(\boldsymbol{y}_{j},\, \boldsymbol{x}_{k},\, \boldsymbol{v}_{k,j}\right). \tag{2.2}
$$

- $\boldsymbol v_{k,j}$：这次观测里的噪声。
- 由于观测传感器形式更多，观测数据 $\boldsymbol z$ 及观测方程 $h$ 也有许多不同形式。称为**观测方程（observation equation）**。

> **本书统一约定补注**：综合时补 $\boldsymbol v_{k,j}\sim\mathcal N(0,\Sigma_v)$。

## §4.4 参数化（Parameterization）—— 两个具体例子（源 §2.3，式 2.3、2.4）

源指出：$f,h$ 与 $\boldsymbol x,\boldsymbol y,\boldsymbol z$ 的具体含义取决于真实运动与传感器种类，存在若干**参数化（Parameterization）**方式。给出两个具体例子。

### 例 1：平面运动 + 位移/角度增量输入（源式 2.3）

设小萝卜在平面中运动，其**位姿**由两个位置和一个转角描述：
$$
\boldsymbol x_k=[x_1,\ x_2,\ \theta]_k^{\mathrm T},
$$
其中 $x_1,x_2$ 是两轴上的位置、$\theta$ 为转角。输入指令是两个时间间隔的位置与转角的变化量：
$$
\boldsymbol u_k=[\Delta x_1,\ \Delta x_2,\ \Delta\theta]_k^{\mathrm T}.
$$
此时运动方程具体化为：
$$
\begin{bmatrix} x_{1}\\ x_{2}\\ \theta \end{bmatrix}_{k}
=
\begin{bmatrix} x_{1}\\ x_{2}\\ \theta \end{bmatrix}_{k-1}
+
\begin{bmatrix} \Delta x_{1}\\ \Delta x_{2}\\ \Delta\theta \end{bmatrix}_{k}
+ \boldsymbol{w}_{k}. \tag{2.3}
$$
源注：「这是简单的**线性关系**。不过，并不是所有的输入指令都是位移和角度的变化量，例如'油门'或者'控制杆'的输入就是速度或加速度量，所以也存在着其他形式更加复杂的运动方程，那时我们可能需要进行动力学分析。」

> **本书统一约定补注**：平面位姿严格属于 $SE(2)$；本章用最朴素的向量加法（$\theta$ 直接相加，未处理角度环绕 $2\pi$）。综合时可指出此为小角度/局部近似，严格写应在 $SE(2)$ 上做群乘法。

**源脚注**：② 位姿（pose）——位置 + 朝向的合称。

### 例 2：二维激光传感器观测一个 2D 路标（源式 2.4）

以小萝卜携带的二维激光传感器为例：观测一个 2D 路标点时能测到两个量——路标点与本体之间的**距离 $r$** 和**夹角 $\phi$**。记
- 路标点 $\boldsymbol y_j=[y_1,\ y_2]_j^{\mathrm T}$，
- 位姿 $\boldsymbol x_k=[x_1,\ x_2]_k^{\mathrm T}$，
- 观测数据 $\boldsymbol z_{k,j}=[r_{k,j},\ \phi_{k,j}]^{\mathrm T}$。

则观测方程为：
$$
\begin{bmatrix} r_{k,j}\\[2pt] \phi_{k,j} \end{bmatrix}
=
\begin{bmatrix}
\sqrt{(y_{1,j}-x_{1,k})^{2}+(y_{2,j}-x_{2,k})^{2}}\\[4pt]
\arctan\!\left(\dfrac{y_{2,j}-x_{2,k}}{y_{1,j}-x_{1,k}}\right)
\end{bmatrix}
+ \boldsymbol{v}. \tag{2.4}
$$

源注：考虑**视觉 SLAM**（传感器是相机）时，观测方程是"**对路标点拍摄后，得到图像中的像素**"的过程——牵涉相机模型，第5讲详述，此处略过。

> **本书统一约定补注**：式(2.4) 把噪声简写为 $\boldsymbol v$，综合时补全为 $\boldsymbol v_{k,j}\sim\mathcal N(0,\Sigma_v)$，$\Sigma_v\in\mathbb R^{2\times2}$。$\arctan$ 严格应为 `atan2`（四象限），原文用 $\arctan$ 是简化。

## §4.5 通用 SLAM 两方程与状态估计的提出（源 §2.3，式 2.5）

保持通用性，把两方程取成抽象形式，SLAM 过程总结为两个基本方程：

$$
\left\{
\begin{array}{ll}
\boldsymbol{x}_{k} = f\!\left(\boldsymbol{x}_{k-1},\,\boldsymbol{u}_{k},\,\boldsymbol{w}_{k}\right), & k=1,\dots,K \\[4pt]
\boldsymbol{z}_{k,j} = h\!\left(\boldsymbol{y}_{j},\,\boldsymbol{x}_{k},\,\boldsymbol{v}_{k,j}\right), & (k,j)\in\mathcal{O}
\end{array}
\right. \tag{2.5}
$$

- $\mathcal O$ 是一个**集合**，记录在哪个时刻观察到了哪个路标（通常不是每个路标在每个时刻都能看到——单个时刻很可能只看到一小部分）。
- **SLAM 基本问题（源原话）**：「当知道运动测量的读数 $\boldsymbol u$，以及传感器的读数 $\boldsymbol z$ 时，如何求解定位问题（估计 $\boldsymbol x$）和建图问题（估计 $\boldsymbol y$）？」
- 由此**把 SLAM 建模成一个状态估计问题**：「如何通过带有噪声的测量数据，估计内部的、隐藏着的状态变量？」

## §4.6 状态估计问题的分类与求解范式（源 §2.3）★ 由此引出滤波 vs 优化

源给出按"方程是否线性、噪声是否高斯"的**四象限分类**与对应方法（逐条保真）：

| 系统类型 | 运动/观测方程 | 噪声分布 | 最优估计方法 | 备注 |
|---|---|---|---|---|
| **线性高斯（LG, Linear Gaussian）** | 线性 | 高斯 | **卡尔曼滤波器（Kalman Filter, KF）** | 最简单；KF 给出**无偏的最优估计** |
| **非线性非高斯（NLNG, Non-Linear Non-Gaussian）** | 非线性 | 非高斯 | **扩展卡尔曼滤波器（EKF）** 与 **非线性优化** 两大类 | 复杂情形；实际 SLAM 多属此类 |

**EKF 求解要点（源原话）**：「我们会在**工作点处把系统线性化**，并以**预测—更新**两大步骤进行求解（见第10讲）。」

**历史脉络（源原话，逐条保真）**：
- 直至 **21 世纪早期**，以 EKF 为主的滤波器方法在 SLAM 中**占据主导地位**。
- **最早的实时视觉 SLAM 系统就是基于 EKF 开发的[2]。**
- 随后为克服 EKF 缺点（**线性化误差**和**噪声高斯分布假设**），人们开始使用**粒子滤波器（Particle Filter）**等其他滤波器，乃至**非线性优化**方法。
- **时至今日，主流视觉 SLAM 使用以图优化（Graph Optimization）为代表的优化技术进行状态估计[13]。**
- **结论（源原话）**：「我们认为优化技术已经明显**优于**滤波器技术，只要计算资源允许，通常都偏向于使用优化方法（见第10讲和第11讲）。」

## §4.7 待澄清的问题（源 §2.3 末，★ 直接引出后续各章）

源在数学表述末尾澄清几个尚未解决的问题，并逐一指向后续章节：

1. **机器人位置 $\boldsymbol x$ 是什么？** 平面可用"两坐标 + 一转角"；但小萝卜多数时候是三维空间机器人。
2. **三维位姿的自由度**：三维运动由 3 个轴构成 → 由"3 个轴上的平移 + 绕 3 个轴的旋转"描述，共 **6 个自由度**。**是否随便用一个 $\mathbb R^6$ 向量就能描述？** 源原话：「我们将发现事情并没有那么简单。对 6 自由度的位姿，如何表达它，如何优化它，都需要一定篇幅来介绍，这将是**第3讲和第4讲**的主要内容。」
3. **视觉 SLAM 中观测方程如何参数化**（空间路标点如何投影到照片）→ **相机成像模型，第5讲**。
4. **怎么求解上述方程** → 需要**非线性优化**知识，**第6讲**。

源原话总结：「这些内容组成了本书数学知识的部分……本讲介绍的内容**构成了本书的提要**。」

**源脚注**：① 6 自由度的位姿（pose）——三维平移 + 三维旋转。

## §5.2 Hello SLAM —— 直接用 g++ 编译（源 §2.4.2）

**程序是什么（源原话）**：在 Linux 中，程序是一个**具有执行权限的文件**，可以是脚本或二进制文件，**不限定后缀名**（不像 Windows 需 `.exe`）。常用的 `cd`、`ls` 等命令就是位于 `/bin` 下的可执行文件；其他地方的可执行程序只要有可执行权限，在终端输入程序名即可运行。

**源代码 `slambook2/ch2/helloSLAM.cpp`（逐字保真）**：
```cpp
#include <iostream>
using namespace std;

int main(int argc, char **argv) {
    cout << "Hello SLAM!" << endl;
    return 0;
}
```

**编译与运行流程（伪步骤/命令，逐字保真）**：

1. 用编译器 `g++`（C++ 编译器）编译：
```bash
g++ helloSLAM.cpp
```
   - 顺利则**无任何输出**。若报 `command not found`，说明未装 g++，安装：
```bash
sudo apt-get install g++
```
   - 若出现别的错误，再检查程序是否输入正确。

2. 编译产物：当前目录多出一个 **`a.out`** 文件（具执行权限，终端里颜色不同）。运行：
```bash
% ./a.out
Hello SLAM!
```

**说明（源原话精炼）**：用编辑器输入 `helloSLAM.cpp` 源码 → 调用 `g++` 编译 → 得可执行文件。`g++` 默认把源文件编译成名为 `a.out` 的程序（可指定输出文件名，留作习题，见 §6 第3题）。此例用了大量默认参数、几乎省略所有中间步骤。下面用 cmake 来编译。

## §5.3 使用 cmake（源 §2.4.3）

**动机（源原话）**：理论上任意 C++ 程序都可用 g++ 编译；但工程规模变大（许多文件夹和源文件、十几个类、复杂依赖、部分编译成可执行文件、部分编译成库），仅靠 g++ 需输入大量编译指令、过程异常烦琐。历史上工程师用 **makefile** 自动编译，而 **cmake** 比它更方便、工程上广泛使用（后面大多数库都用 cmake 管理源代码）。

**cmake 工作机制（源原话）**：用 `cmake` 命令生成一个 **makefile** 文件，再用 `make` 命令据该 makefile 编译整个工程。

**`slambook2/ch2/CMakeLists.txt`（最小工程，逐字保真）**：
```cmake
# 声明要求的cmake最低版本
cmake_minimum_required( VERSION 2.8 )
# 声明一个cmake工程
project( HelloSLAM )
# 添加一个可执行程序
# 语法：add_executable( 程序名 源代码文件 )
add_executable( helloSLAM helloSLAM.cpp )
```
说明：`CMakeLists.txt` 告诉 cmake 对这个目录下的文件做什么，内容需遵守 cmake 语法。此示例演示最基本工程：指定一个工程名 + 一个可执行程序。

**编译流程 A（在源目录直接编译，伪步骤，逐字保真）**：

1. 在当前目录（`slambook2/ch2/`）调用 cmake：
```bash
cmake .
```
   cmake 输出编译信息，在当前目录生成中间文件，其中最重要的是 **MakeFile**（自动生成，不必修改）。
2. 用 make 编译：
```bash
% make
Scanning dependencies of target helloSLAM
[100%] Building CXX object CMakeFiles/helloSLAM.dir/helloSLAM.cpp.o
Linking CXX executable helloSLAM
[100%] Built target helloSLAM
```
3. 得到可执行程序 `helloSLAM`，执行：
```bash
% ./helloSLAM
Hello SLAM!
```

**cmake vs g++ 的区别（源原话精炼）**：cmake 过程处理工程文件之间的关系，make 过程实际调用 g++ 编译程序。虽多了 cmake、make 两步，但项目编译管理从"输入一串 g++ 命令"变成"维护若干直观的 CMakeLists.txt 文件"，明显降低维护难度（新增可执行文件只需加一行 `add_executable`，cmake 自动解决依赖关系）。

**编译流程 B（推荐：build 目录外部构建，伪步骤，逐字保真）**：
源指出流程 A 的不满：cmake 中间文件留在代码目录里，发布时需逐个删除、不便。更好做法是把中间文件放进一个中间目录，编译成功后删该目录。**更常见做法**：
```bash
mkdir build
cd build
cmake ..
make
```
说明：新建中间文件夹 `build`，进入后用 `cmake ..` 对上一层（代码所在文件夹）编译；cmake 中间文件生成在 `build` 内，与源代码分开。发布源代码时只要删掉 `build` 文件夹即可。

## §5.4 使用库（源 §2.4.4）

**库是什么（源原话）**：C++ 工程中并非所有代码都编译成可执行文件——**只有带 `main` 函数的文件才生成可执行程序**；另一些代码只想打包供其他程序调用，这个东西叫**库（Library）**。库往往是许多算法、程序的集合（如 **OpenCV** 提供计算机视觉算法、**Eigen** 提供矩阵代数计算）。

**自写库源文件 `slambook2/ch2/libHelloSLAM.cpp`（逐字保真）**：
```cpp
//这是一个库文件
#include <iostream>
using namespace std;

void printHello() {
    cout << "Hello SLAM" << endl;
}
```
该库提供 `printHello` 函数；它**没有 main 函数**，意味着库中没有可执行文件。

**在 CMakeLists.txt 中编译为静态库**：
```cmake
add_library( hello libHelloSLAM.cpp )
```
告诉 cmake 把此文件编译成名为 `hello` 的库。编译（伪步骤）：
```bash
cd build
cmake ..
make
```
→ `build` 文件夹中生成 **`libhello.a`** 文件（即得到的库）。

**静态库 vs 共享库（源原话，对照表）**：

| 库类型 | 后缀 | 调用行为 | 空间 |
|---|---|---|---|
| 静态库（static） | `.a` | 每次被调用都生成一个**副本** | 占空间多 |
| 共享库（shared） | `.so` | 只有**一个副本** | 更省空间 |

生成**共享库**而非静态库，只需：
```cmake
add_library( hello_shared SHARED libHelloSLAM.cpp )
```
→ 得到 **`libhello_shared.so`**。

**头文件 `slambook2/ch2/libHelloSLAM.h`（逐字保真）**：
源原话：库文件是一个压缩包，里面是编译好的二进制函数；仅有 `.a`/`.so` 库文件并不知道里面的函数是什么、调用形式如何。为让别人（或自己）使用，需提供**头文件**说明库里有什么。库的使用者只要拿到头文件 + 库文件即可调用。
```cpp
#ifndef LIBHELLOSLAM_H_
#define LIBHELLOSLAM_H_
// 上面的宏定义是为了防止重复引用这个头文件而引起的重定义错误
// 打印一句Hello的函数
void printHello();
#endif
```

**调用库的可执行程序 `slambook2/ch2/useHello.cpp`（逐字保真）**：
```cpp
#include "libHelloSLAM.h"
// 使用libHelloSLAM.h中的printHello()函数
int main(int argc, char **argv) {
    printHello();
    return 0;
}
```

**在 CMakeLists.txt 中生成可执行程序并链接到库（逐字保真）**：
```cmake
add_executable( useHello useHello.cpp )
target_link_libraries( useHello hello_shared )
```
说明：通过这两行，`useHello` 程序就能顺利使用 `hello_shared` 库中的代码。对他人提供的库也可用同样方式调用、整合进自己的程序。

**小结（源原话，逐条保真）**：
1. 程序代码由**头文件**和**源文件**组成。
2. 带有 `main` 函数的源文件编译成**可执行程序**，其他的编译成**库文件**。
3. 可执行程序若想调用库文件中的函数，需**参考该库提供的头文件**以明白调用格式，同时要把可执行程序**链接到库文件**上。

源附问（→ 习题第6题）：若代码引用了库函数却忘了链接，会发生什么？试着把 CMakeLists.txt 链接部分去掉看报错。

**源脚注**：① Linux 中库文件分静态库（`.a`）与共享库（`.so`）两种。
